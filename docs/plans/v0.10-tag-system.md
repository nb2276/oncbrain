# CEO Plan: Cross-Cutting Tag System

Generated by /plan-ceo-review on 2026-05-30
Branch: main (pre-feature-branch) | Mode: SCOPE EXPANSION
Repo: nb2276/oncbrain

## Original ask
"Create a hashtag system on top of the disease site. Add to bottom of left nav.
Ideas include radiation, palliation, meeting tag, study conclusion (early,
confirmatory, practice changing, etc.)"

## Reframe (after Step 0 premise challenge)

Two of the four proposed namespaces ALREADY EXIST as structured features:
- **"Study conclusion"** is `verdict.soc_implication` (`src/lib/verdict.ts`),
  with rationale + audience fields that drive the verdict pill UI.
- **"Meeting tag"** is `conferences/[slug]/` landing pages, indexed off the
  per-digest `conference` field.

Genuinely new from the original ask: **modality** (radiation, surgery, systemic,
combined) and **intent** (curative, palliative, supportive). Codex's outside
voice challenge then surfaced that absorbing verdict and conferences into a
flat tag namespace would lose structured rendering (verdict rationale/audience)
and duplicate date metadata onto every study.

**Resolved architecture:** unified tag *namespace* lives as a view/index layer.
Each axis keeps its structured field on the underlying record (`study.verdict`
stays a typed object, `digest.conference` stays per-digest). Tags like
`verdict:confirmatory` and `meeting:asco-2026` are computed FROM those fields
at build time. Genuinely new namespaces (modality, intent, methodology) get
new fields on `DigestStudy`. Every namespace is queryable through a single
tag interface even though the data layer remains typed.

## Vision

### 10x check
oncbrain becomes the canonical taxonomy layer for oncology meeting research.
Every study sits at the intersection of disease-site × verdict × modality ×
intent × meeting × methodology, and a reader can land on
`/tags/modality:radiation+intent:palliation/` and see the sibling studies in
that exact slice. The tag layer drives navigation, RSS, search, the JSON API,
and (eventually) per-subspecialty email subscriptions — one taxonomy, every
surface.

### Platonic ideal
Walking onto oncbrain feels like a peer subspecialist already pre-read every
paper and shelved it for you. You don't search — you wander. A study card
says "Confirmatory · RT · palliative · ASCO 2026" and the page shows you the
3 other studies in that exact intersection from the last 6 months. The
taxonomy doesn't just organize; it generates the next read.

### Corpus reality (per Codex)
Current corpus is **9 digests, 41 studies** (verified via
`for f in data/digests/*.json; do jq '[.digest.sites[]?|.studies[]?]|length'`).
At v1, the 3-way intersection architecture is mostly dormant — expect 0-5
intersection pages actually generated. As corpus grows (every digest adds
~4.5 studies → ~100 studies projected by end of 2026), intersection pages
light up organically without code change. CEO decision: build the architecture
now, accept it's mostly dormant in v1. Cathedral posture acknowledged.

## Accepted Scope (v1)

### Data layer
- `DigestStudy.tags`: optional array of `{ namespace: string; value: string }` objects.
- New structured fields on `DigestStudy`: `modality`, `intent`, `methodology` (enum-typed; null when not applicable).
- `src/lib/tags.ts`: namespace enums + canonicalization + reverse-index builder.
- `src/lib/llm-pipeline.ts`: Phase 2 prompt emits the new fields.
- `prompts/digest-v5-study-agent.txt`: extended with namespace definitions + enum values + emit instructions. VOICE.md already cached, so token cost is one-shot.

### Build pipeline
- Reverse-index builder reads every `data/digests/*.json`, emits a tag-to-studies map plus an intersection-cardinality table.
- Cardinality guard: 3-way max, N≥3 threshold, hard cap 1000 (sorted desc by count).
- Eval rubric extended in `prompts/eval-judge-v1.txt` with tag-namespace consistency criteria.
- Build log emits per-namespace emission counts and dropped-invalid-tag counts.

### URL surface
- `/tags/<ns>:<slug>/` — single-tag landing pages
- `/tags/<a>+<b>/` — 2-way intersection (alphabetical canonical ordering, 301 redirect from non-canonical)
- `/tags/<a>+<b>+<c>/` — 3-way intersection (only emitted when N≥3)
- `/tags/<...>/feed.xml` — per-tag/intersection RSS feed
- `/api/v1/tag/<ns>:<slug>.json` — per-tag JSON API
- `/tags/` — top-level tag index page (linked from header sub or footer)

### UI
- Tag chips render at the bottom of TriageRail on desktop (≥1200px) under a "Tags" subhead, smaller font + muted color so they don't compete with the primary disease-site nav.
- Mobile: tag chip row below the existing disease-site chip row on home.
- Per-study tag chips render on `StudyCard.astro` head — limit visible chips to disease-site + verdict + 1-2 feature tags to avoid chip overload; rest fold into the depth section.
- Tooltips: hover on desktop; tap-to-toggle bottom-sheet on mobile.
- Breadcrumb chain on intersection pages: `breast › modality:radiation › verdict:practice-changing` with deletable links.
- "Studies like this" footer at the bottom of each study card's depth fold: up to 3 siblings sharing primary disease-site + verdict + dominant modality.
- 30-day tag-activity histogram on home: small sparkline strip listing top-5 tags by recency-weighted study count.

### Backfill (operational)
- `npm run build:day -- --backfill` re-runs the full pipeline on all 9 existing digests. Accepted side effects: TL;DR / verdict / slug / Obsidian export changes vs published versions.
- Backfill is a one-time operation pre-deploy; commit the regenerated artifacts in a separate `chore(content): backfill tags across digests` commit so the diff is auditable.

### Tests
- `test/tags.test.ts` — canonicalization, validation, reverse-index correctness
- `test/intersection-build.test.ts` — cardinality cap, N≥3 threshold, alphabetical canonical ordering
- `test/pages-tags.test.ts` — getStaticPaths emits the expected URL set
- `test/publish-boundary.test.ts` — extended: assert no `/tags/` page surfaces a filed PDF
- `test/feed-tag.test.ts` — per-tag RSS feed correctness
- `test/api-tag.test.ts` — per-tag JSON shape, no PDF refs
- `test/eval-tag-consistency.test.ts` — LLM-as-judge harness for tag-namespace consistency

## Error & Rescue Registry

| Method/Codepath | Failure mode | Rescue action | User sees |
|---|---|---|---|
| Phase 2 emits invalid tag value | LLM hallucination outside enum | Drop tag, log warning with study slug, retain other tags | Nothing (transparent); curator sees build log |
| Phase 2 emits empty tag set for a namespace | Missing classification | Study tagged for available namespaces; doesn't appear on the missing-namespace landings | Subtly missing from one landing page; curator alert for methodology specifically |
| `data/digests/<date>.json` corrupted on read | I/O / parse failure | Skip date in tag index build; remaining digests still indexed | Sparse landings for that day; build log error |
| Intersection cardinality exceeds 1000 | Build-time overrun | Sort by study count desc, take top 1000, log dropped | Some long-tail intersections 404; build log alert |
| Phase 2 LLM call fails entirely | Network / quota | Inherits existing Phase 2 retry behavior (`AnthropicLlmClient`); after retries, study has no tags | Missing from all tag landings; curator notified per existing build-failure handling |
| Slug not in regex `[a-z0-9-]+` | Malformed value | Reject at canonicalization; never reaches URL surface | Nothing (defense-in-depth) |
| Non-canonical intersection URL order | Reader hits `/tags/b+a/` instead of canonical `/tags/a+b/` | 301 redirect to canonical | Brief redirect; bookmark resolves correctly |
| Intersection page existed yesterday, dropped below N=3 today | Reader clicks stale bookmark | 410 Gone stub from `removed-intersections.json` (30-day retention) | "This intersection is no longer published" message; link to closest single-tag page |
| LLM emits structurally valid but clinically wrong tag (palliative misclassified as curative) | Semantic miscall | Eval rubric flags via judge LLM in CI; curator override via `npm run override --tag-edit` | Wrong content on landing pages until override or rebuild |

### Deployment sequence
```

2. Run npm test (target 100% pass)
3. Run npm run eval (tag-quality eval rubric extension)
4. Local: npm run build:day -- --backfill (regenerates all 9 digests)
5. Inspect diff: VERIFY TL;DR / verdict drift acceptable
6. Commit content backfill separately from feature code
7. /ship the feature branch
8. DO auto-deploys
9. Post-deploy verification:
   - /tags/ index page renders
   - Spot-check 3 single-tag landings
   - Spot-check 1 intersection page (if any exist)
   - /feed.xml + /api/v1/digests.json still valid (unchanged)
   - Existing slugs (/2026-05-27/#senomac) still resolve
```

### Rollback flowchart
```
DEPLOY FAILS / WRONG RESULT
  │
  ├─[code-only issue]─▶ git revert feature commit ─▶ /ship revert
  │
  ├─[content backfill caused wrong studies]─▶ git revert content commit ─▶ /ship revert
  │  (preserves the feature code if it was deployed earlier)
  │
  └─[both]─▶ revert content first (cheap), then code
```

## Reviewer Concerns

After the multi-section review + Codex outside voice, these gaps remain that
the user accepted into v1 with eyes open:

1. **Backfill slug-stability risk:** `build:day --backfill` may regenerate
   different slugs (e.g. via `deriveSlug`'s clustering) versus the published
   versions. Reader bookmarks to `/2026-05-27/#senomac` could break if the
   slug derivation drifts. CEO accepted on "full consistency" basis. If
   observed in practice, add a `slug-redirects.json` artifact as a follow-up.

2. **Intersection page sparsity at v1 launch:** With 41 studies, expect 0-5
   intersection pages actually generated. The architecture is real, the cap
   is real, but the reader experience is single-tag landings for v1. CEO
   accepted on cathedral posture (architecture ready when corpus grows).

3. **Tag quality is the product, not parser hygiene:** Eval rubric extension
   catches some semantic miscalls; curator override handles individual fixes.
   No structural mechanism prevents the LLM from drifting on hard semantic
   calls (palliative vs curative borderline cases).

4. **Left rail surface contention:** Codex correctly noted that TriageRail
   is intra-page and tag nav is cross-corpus. User explicitly requested left
   rail in original ask. /plan-design-review should pixel-mock this before
   implementation and may push the tag section out of the rail entirely.
