# v0.5 — Multi-source ingestion (LOCKED 2026-05-18)

Two cross-source reviews (initial codex + revised codex post-feedback). All 3 P0s and 3 P1s folded in. PDFs deferred to v0.6+.

## Goal

Tweets (existing), PubMed papers (new), and slide photographs (new) all flow through the v0.4 3-pass pipeline. Cross-source clustering when the same trial appears in multiple inputs. Per-claim source attribution in Phase 2 to prevent mixed-source hallucination.

## Universal Telegram inbox

The existing `@oncbrain_bot` becomes the single ingestion channel.

```
incoming Telegram message → save to inbox_items IMMEDIATELY
─────────────────────────────────────────────────────────────────
twitter.com/x.com URL          type='tweet'
pubmed.ncbi.nlm.nih.gov URL OR  type='paper'
  raw "PMID: \d+" citation
image attachment               type='slide'
free text                      attached as curator_note on the
                               nearest typed item in same message
```

**Critical: offset advances only after inbox write succeeds.** This addresses codex P0 #1 (lossy queue). v0.5 NEVER does enrichment (NCBI fetch, OCR) inline during pull:telegram. Enrichment is a separate async pass that can retry indefinitely without losing the source message.

## Enrichment loop (async, idempotent, retryable)

New `npm run enrich:inbox` script processes `inbox_items WHERE enrichment_status IN ('pending', 'failed')`:

```
For each row:
  switch (row.type) {
    case 'tweet':
      → call fetchTweet() (existing oEmbed flow)
      → write bookmarks row, set enriched_row_id
    case 'paper':
      → call pubmedClient.fetch(row) → metadata + abstract + PMC excerpt
      → if 503/network failure: mark 'failed', leave row for next pass
      → write papers row
    case 'slide':
      → call slidePhotoStorage.process(row) → OCR + hash
      → write slide_uploads row
  }
```

Retry policy: failed rows automatically retry on the next enrich run (up to 5 attempts, then mark 'deferred' for manual review). Telemetry logs the failure for the curator.

The 3am cron runs `pull:telegram && enrich:inbox && build:day yesterday today && build && git push`.

## PubMed enrichment

`src/lib/pubmed-client.ts` wraps NCBI E-utilities:

1. `esummary.fcgi?db=pubmed&id=<PMID>&retmode=json` → metadata, MeSH, PMC ID
2. If PMC ID present → `efetch.fcgi?db=pmc&id=<PMC>&retmode=xml` → parse body
3. Section-filter extracted body: keep Methods + Results, drop Introduction/Discussion/References
4. Token-cap the filtered text to ~2000 tokens (codex revised down from my proposed 8000)
5. If full text unavailable: store `pmc_id=null` and emit degraded-state flag

3 req/sec without API key — well within solo-curator volume.

## Slide photo enrichment

`src/lib/slide-photo-storage.ts`:

1. Download Telegram photo via getFile API
2. Validate file is image/jpeg or image/png (mime + magic-bytes check)
3. Generate filename: `data/slide-photos/<bookmark_date>/<uuidv4>.jpg`
4. Run Apple Vision OCR (existing binary)
5. Compute sha256 hash for cache stability (v0.4 pattern)
6. Write slide_uploads row with source_batch_key derived from telegram_msg_id

## Image trust model (separate path for slides)

**Slides are NEVER served with the pbs.twimg.com host allowlist path.** Codex P1 #5: punching a hole in the v0.4 SSRF defense would defeat the whole hardening.

Astro adds a new static route `/slides/[file_path].jpg` that serves from `data/slide-photos/`. Filenames are UUIDs so guessing is infeasible. The route validates the path is under `data/slide-photos/` and rejects anything else.

A new `isSafeSlideUrl()` check accepts only `/slides/` prefix URLs at render time. Tweet figures keep the existing `isSafeImageUrl()` (pbs.twimg.com only).

## Association (weighted, not boolean)

`src/lib/source-association.ts`:

```ts
type AssociationEdge = {
  itemA: string;  // "tweet:42" | "paper:pmid12345" | "slide:uuid"
  itemB: string;
  weight: number; // see scoring below
  reason: string; // "shared NCT04567890" etc.
};
```

Scoring:
- **+10 (strong)**: two items share an exact NCT ID (validated `NCT\d{8}` regex match in both)
- **+3 (medium)**: an exact trial acronym appears in 2+ items (recall: acronym is normalized — uppercase, hyphenless — and must be in a curated whitelist OR appear in source text outside of figure OCR)
- **0 (ignore)**: single-acronym occurrence in noisy OCR alone

The association graph is presented to Phase 1 as a SOFT hint: "items A, B, and C share NCT04567890 — strong clue they describe the same trial. Cluster together unless content clearly contradicts."

**No hard pre-groups.** Codex P0 #3: hard hints + Phase 1's partition validator would force the model to preserve wrongful merges, then Phase 2 hallucinates from heterogeneous evidence.

## Pipeline updates (Phase 2 source-tagged claims)

`src/lib/llm-pipeline.ts`:

```ts
type DigestInputItem =
  | DigestInputTweet
  | DigestInputPaper
  | DigestInputSlide;

type SourceTaggedClaim = {
  claim: string;               // the bullet text
  evidence_sources: Array<{    // which sources support this claim
    source_type: 'tweet' | 'paper' | 'slide';
    source_id: number | string;
    quote?: string;            // optional evidence quote
  }>;
};

type DigestStudy = {
  // existing fields...
  details: string[];           // unchanged externally
  // INTERNAL only — used by validator, not rendered:
  details_with_sources?: SourceTaggedClaim[];
};
```

Phase 2 prompt updated to emit `details_with_sources` per bullet. A new validator (`validateClaimSources`) checks each numeric token in a claim against the OCR/abstract of the cited source. Mismatches drop the claim (existing v0.4 caption-validator pattern, extended).

Render layer continues to consume the flat `details: string[]` for backward compat. Source-tagged claims are internal hardening only in v0.5.

## Rendering (Astro)

Each study card gets a **source attribution row** above the figure:

```
🐦 5 tweets · 📄 1 paper · 🩻 2 slides
```

Click "📄 1 paper" → expands to show PMID/DOI badge + abstract toggle.
Click "🩻 2 slides" → expands to show standalone slide images with source_label captions.

Standalone paper studies (no associated tweets): render the abstract toggle + PMID link as the source list. Optional PMC link if available.

Standalone slide studies (e.g., curator captured a slide from a talk not on Twitter): render the slide image full-width with source_label.

Mixed clusters: all three source types appear, ordered tweets → paper → slides.

## Open questions for implementation time

1. How to handle a Telegram message containing BOTH a tweet URL AND a PubMed citation block (rare but possible)? Probably: create two inbox_items, both back-referencing the same telegram_msg_id.
2. When an `inbox_items` row's enrichment retries exhaust (5 attempts), how should the next `build:day` treat it? Option A: skip silently. Option B: emit a degraded-state disclosure. Option B is consistent with v0.4's "surface, don't silently omit" principle.
3. For the acronym whitelist: should it be hardcoded in extract.ts or live in a curator-editable JSON file? Hardcoded for v0.5; revisit if it needs frequent updates.
