# v0.13 — Trials to watch for each open question

For each open question on a study card, list the trials in progress that watch that question. Trials may be drug, radiation, surgical, biomarker, sequencing, or methodological, whatever the question is actually about. Each trial is paired structurally to the specific open question it watches. The card surfaces a small "N trials watching" badge in the depth-fold summary so the feature is discoverable on mobile even when the fold is collapsed. Goal: show readers, per question, the next trial to watch.

Status: planned (v3 after open-question redirect + codex round 2). No code written. `/plan-eng-review` + 2 codex outside-voice passes completed 2026-06-08 against a clean tree on `main`.

## Decisions locked

| # | Decision | Choice |
|---|---|---|
| 1 | Data source | Hybrid: Phase 2 LLM emits per-open-question search queries; for each query, CT.gov v2 API returns candidates; LLM reranks across all candidates, pairing each pick to the open question it watches |
| 2 | Pipeline interleaving | Inside the per-study `runConcurrent` callback, after `runStudyAgent` succeeds and BEFORE `successful.push`: fan out CT.gov fetches (one per query, deduped by in-run cache, global fetch semaphore=3), aggregate, single rerank LLM call, attach `related_trials` to the study. Orchestrator runs in its own try/catch INSIDE the existing Phase 2 try; enrichment exceptions never drop the study (codex #5) |
| 3 | Failure mode | Silent skip per branch with differentiated logs. End-of-build minimum-success gate: WARN if eligible-and-populated ratio drops below 50%, where eligible includes BOTH studies-with-related_search AND studies-without (broken out separately so retrieval-wide outages can't game the metric — codex #15) |
| 4 | Time semantics | Live pipeline. Every rebuild re-fetches CT.gov. A 2024 digest rebuilt in 2026 shows 2026 statuses |
| 5 | In-build cache | In-memory object passed THROUGH `runBackfill` so a multi-date backfill shares one cache across `buildOneDate` calls (codex #20). Single-date builds get a fresh cache scoped to that call. Cache stores resolved results NOT promises; failures are NOT cached, preventing one transient failure from poisoning later identical queries (codex #13). Keyed on sha1(normalize(query.term)). No disk persistence; no `--use-cache` flag |
| 6 | Scope | Per-study only. No site-level aggregation |
| 7 | Candidate richness (codex #3) | CT.gov fetch returns NCT, BriefTitle, OverallStatus, Phase[], EnrollmentCount, PrimaryCompletionDate, **BriefSummary, Conditions[], Interventions[] (name + type), EligibilityCriteria** (brief inclusion/exclusion text). Rerank sees real content, not just title; this is the data that lets the LLM judge fit and write grounded relevance phrases |
| 8 | Status filter (codex #12) | RECRUITING + NOT_YET_RECRUITING + ACTIVE_NOT_RECRUITING + ENROLLING_BY_INVITATION. Captures near-readout trials (active, follow-up phase) which are the most-relevant "watch" candidates. Per-trial status emoji distinguishes state on the card |
| 9 | Field shape per displayed trial | NCT, brief_title, overall_status, phase[] (may be multi e.g. PHASE1\|PHASE2), enrollment_count (nullable; relabel: CT.gov field is current enrollment count, not contracted target), primary_completion_date (nullable, YYYY-MM, labeled "primary completion" NOT "readout" — codex #7), answers_question (exact source string from `study.open_questions`), relevance_phrase (<= 60 char LLM tie, grounded in the candidate's BriefSummary/Eligibility/Interventions, not invented — codex #4). Sort within each question group by primary_completion_date ascending (nulls last) |
| 10 | Card layout | **Grouped, flat-div nest (not nested `<ul>`).** Each question's `<li>` gets a sibling `<div class="trials-watching" role="group" aria-label="Trials watching this question">` rendered below the question text. Visual nesting via `border-left: 2px solid var(--border)` on `.trials-watching`, zero left-indent (preserves 375px horizontal budget; resolves codex #27 structurally). Each `.trials-watching > div` row has `min-height: 44px; display: flex; align-items: center` (touch target per DESIGN.md). Questions WITH watching trials get `font-weight: 600` so the visual irregularity (some questions have nests, others don't) reads as intentional. Depth-fold `<summary>` gains a `<span class="trials-badge" aria-label="N trials watching"> · N trials watching</span>` segment after the existing count text, color `var(--fg-muted)`, singular/plural correctly, hidden at zero, hidden at viewport ≥1024px (depth fold auto-expands; badge redundant) |
| 11 | Eval scope | New dimensions: `related_query_targeting` (judges whether queries TARGET each open question, not just wording), `related_trial_relevance` (judges whether the picked trial ACTUALLY ADDRESSES the watched question, using BriefSummary + Eligibility text not just title — codex #32). ## Data flow

```
                         Phase 1 grouping (unchanged)
                                    │
                                    ▼
                   Phase 2 study-agent loop  (runConcurrent, 4-wide)
                       │
        runStudyAgent ─┤  REFACTOR: returns { study, raw } (codex #18) so the
                       │  orchestrator can read related_search off the raw
                       │  response. parseStudyAgentResponse still produces the
                       │  same DigestStudy shape; related_search is NEVER
                       │  copied onto study.
                       │
                       │  Phase 2 try wraps:
                       │    runStudyAgent → enrichStudyWithRelatedTrials
                       │  enrichStudyWithRelatedTrials has its own internal
                       │  try/catch; ANY exception inside enrichment sets
                       │  related_trials=null + logs, NEVER throws (codex #5).
                       ▼
        enrichStudyWithRelatedTrials(study, raw, deps, backfillCache)
              │
              │  related_search present + queries pass stop-list?
              │     no  → study.related_trials = null
              │            log "no_related_search" + provenance.rerank_outcome="skipped"
              │     yes ▼
              │
              │  for each query in raw.related_search.queries (cap 3):
              │     normalized_key = sha1(normalize(query.term))
              │     backfillCache.has(normalized_key) ?
              │        yes → reuse cached candidates (may include "empty" sentinel)
              │        no  → acquire fetch semaphore (global, limit 3 — codex #14)
              │              await deps.ctgovFetch(query.term)
              │              if ok → backfillCache.set(normalized_key, candidates)
              │              if fail → do NOT cache; release semaphore
              │              (failures NEVER stored — codex #13)
              │
              │     CT.gov request:
              │       GET clinicaltrials.gov/api/v2/studies
              │         ?query.term=<term>
              │         &filter.overallStatus=RECRUITING|NOT_YET_RECRUITING|
              │                               ACTIVE_NOT_RECRUITING|ENROLLING_BY_INVITATION
              │         &pageSize=20
              │         &fields=NCTId,BriefTitle,OverallStatus,Phase,
              │                 EnrollmentCount,PrimaryCompletionDate,
              │                 BriefSummary,Conditions,Interventions,
              │                 EligibilityCriteria
              │       ├── 5xx → retry max 3, exp backoff via deps.sleep
              │       ├── 429 → fail this query, log "ctgov_rate_limit"
              │       ├── timeout 10s → log "ctgov_timeout"
              │       └── 0 results → ok:false, error:"empty", log "ctgov_empty"
              │
              │  Each successful query produces CandidateTrial[], tagged with
              │  the question it was sought for. Provenance records query terms
              │  fired and failed.
              │
              │  zero successful queries:
              │     study.related_trials = null
              │     provenance.rerank_outcome = "skipped" (no candidates to rank)
              │     log "all_queries_failed"
              │
              │  ≥ 1 successful query:
              │     try deps.rerankRelatedTrials(grouped_candidates, study.open_questions, study.tldr)
              │
              │  rerank prompt template: prompts/digest-v5-rerank-trials.txt
              │  with VOICE.md cached. Instructs LLM:
              │     ─ Inputs: per-question candidate pools, each candidate
              │       includes BriefSummary + Conditions + Interventions +
              │       Eligibility, so fit is judgeable from real content.
              │     ─ Pick UP TO 5 across all pools.
              │     ─ Each pick declares answers_question = EXACT string
              │       from study.open_questions (rerank forbidden to invent).
              │     ─ Each pick's relevance_phrase (≤ 60 chars) must ground
              │       in observable candidate fields (a primary endpoint, an
              │       inclusion criterion, an intervention name). Phrase
              │       MUST NOT invent endpoints or populations.
              │     ─ Distribute picks across questions when possible; depth
              │       beats breadth only when other questions truly have no
              │       strong candidate.
              │     ─ Abstain freely: 0 picks is allowed and preferred over
              │       weak picks.
              │
              │  rerank malformed JSON: retry once, then null
              │  rerank exception: null, provenance.rerank_outcome="failed",
              │                    fallback to deterministic-degraded:
              │                    top trial per question by primary_completion
              │                    _date ascending, with relevance_phrase=
              │                    "candidate match" (no LLM tie). Capped 1
              │                    per question, ≤ 3 total. Logged
              │                    "rerank_failed_fallback_used" (codex #17)
              │
              │  parseRelatedTrials(rerank.raw OR fallback, candidates,
              │                     study.open_questions)
              │     ─ re-hydrate full CandidateTrial fields by nct
              │     ─ INVARIANT 1: every picked nct ∈ aggregated candidate set
              │     ─ INVARIANT 2: every answers_question ∈ study.open_questions
              │     ─ DEDUP: (nct, answers_question) tuple unique (codex #21)
              │     ─ DEDUP: same nct under different questions allowed (a single
              │       trial can legitimately watch multiple questions; informs
              │       cross-question signal)
              │     ─ cap 5 total
              │     ─ group by answers_question, preserving study.open_questions order
              │     ─ within each group, sort by primary_completion_date asc
              │     ─ status enum check (filter set above)
              │     ─ skip phase chip rendering when phase[] is empty/null
              │       (codex #30 — no "—" chip)
              │     ─ return RelatedTrial[] | null
              │
              ▼
        study.related_trials = RelatedTrial[] | null
        study.related_trials_provenance = { … } | null
        successful.push({ cluster, study })  ← unchanged ordering
                                    │
                                    ▼
                         Phase 3 synthesis (unchanged)
                                    │
                                    ▼
                         applyDigestOverrides() — new fields plumbed with
                         INVARIANT re-check after Phase 2 question text change
                                    │
                                    ▼
                         data/digests/<date>.json (includes provenance)
                                    │
                                    ▼
                         astro build → StudyCard.astro renders:
                           ─ "N trials watching" badge in depth-fold summary
                           ─ Nested trials inside .open-questions <ul>
```

## Type contract

```ts
// src/lib/digest-data.ts AND src/lib/llm-pipeline.ts (mirrored)

// Pre-rerank shape. NOT persisted in the artifact.
// Includes the richer codex-#3 fields so rerank can judge fit.
export type CandidateTrial = {
  nct: string;
  brief_title: string;
  overall_status:
    | 'RECRUITING'
    | 'NOT_YET_RECRUITING'
    | 'ACTIVE_NOT_RECRUITING'
    | 'ENROLLING_BY_INVITATION';
  phase: string[] | null;             // codex #11: CT.gov returns array
  enrollment_count: number | null;    // codex #11: current count, not target
  primary_completion_date: string | null;  // YYYY-MM trimmed
  brief_summary: string | null;       // ~500 chars, CT.gov BriefSummary
  conditions: string[];               // CT.gov Conditions
  interventions: Array<{
    name: string;
    type: string;  // 'DRUG' | 'RADIATION' | 'PROCEDURE' | 'DEVICE' | 'BIOLOGICAL' | etc.
  }>;
  eligibility_brief: string | null;   // first ~300 chars of EligibilityCriteria
};

// Post-rerank, paired to one of study.open_questions.
// What we attach to the study and ship in the artifact.
export type RelatedTrial = CandidateTrial & {
  // Exact string from study.open_questions (INVARIANT-validated).
  answers_question: string;
  // ≤ 60 chars. LLM-written tie, grounded in candidate fields.
  // VOICE.md compliant via the shared cache block.
  relevance_phrase: string;
};

// Auditable provenance — codex #36.
// Not rendered; ships in /api/v1/*.json for downstream consumers.
export type RelatedTrialsProvenance = {
  queries_fired: string[];
  queries_failed: Array<{ term: string; reason: string }>;
  candidates_returned: number;
  fetched_at: string;  // ISO timestamp
  rerank_outcome: 'picked_N' | 'abstained' | 'failed' | 'skipped' | 'fallback';
};

export type DigestStudy = {
  // ...existing fields...

  // v0.13: trials watching each open question.
  related_trials?: RelatedTrial[] | null;
  related_trials_provenance?: RelatedTrialsProvenance | null;
};

// Phase 2 prompt addition (prompts/digest-v5-study-agent.txt):
//   related_search: {
//     queries: [
//       { term: string,            // CT.gov free-text query
//         watches_question: string // EXACT text from study.open_questions[i]
//       },
//       ... // up to 3
//     ]
//   } | null
// Rules:
//   ─ For each open question, decide if there are likely to be open trials
//     in progress addressing it. If yes, emit a free-text query.
//   ─ Queries may target drug, radiation, surgery, biomarker, sequencing,
//     or methodology. Match the question's nature.
//   ─ Cap at 3; pick the highest-leverage open questions if more than 3.
//   ─ query.term must be specific enough to retrieve relevant matches; the
//     stop-list rejects single-word generics, bare drug names without context,
//     and queries shorter than 3 words.
//   ─ Emit null when no open question maps cleanly to a CT.gov query.
```

## Retrieval framing (final)

Per-open-question queries. `query.term` (free-text) covers drug, RT, surgery, biomarker, methodology generically. Codex correctly noted (#8) that recall varies across domains; the eval dimension `related_query_targeting` is the regression signal if any domain underperforms, and the cross-domain fixture set (codex #34) catches this at baseline.

Codex #10 (no guardrails): the stop-list catches obvious junk; for plausible-but-badly-scoped queries, the eval dimension is the catch. We do NOT add age/stage/line-of-therapy guardrails to the query itself; CT.gov's `query.term` ranking handles broad recall, and the rerank LLM (with BriefSummary + Eligibility in hand) is the precision filter.

Codex #9 (pageSize=20 no pagination): accepted as a v0.13 limit. If eval signals trials are missed beyond page 1, paginate in v0.13.1. Track via the `candidates_returned` provenance field.

## Failure modes

| Branch | Log line | Provenance | Reader sees |
|---|---|---|---|
| Phase 2 emits no `related_search` | `[v0.13] no_related_search: <slug>` | rerank_outcome="skipped" | no badge, no nest |
| Phase 2 emits empty queries array | `[v0.13] empty_queries: <slug>` | rerank_outcome="skipped" | no badge, no nest |
| Query term fails stop-list | `[v0.13] junk_query: <slug> · <term>` | logged in queries_failed | other queries proceed |
| CT.gov 5xx for a query after 3 retries | `[v0.13] ctgov_fetch_failed: <slug> · <term> · <error>` | queries_failed | that query dropped |
| CT.gov 429 for a query | `[v0.13] ctgov_rate_limit: <slug> · <term>` | queries_failed | that query dropped |
| CT.gov timeout (10s) | `[v0.13] ctgov_timeout: <slug> · <term>` | queries_failed | that query dropped |
| CT.gov 0 results | `[v0.13] ctgov_empty: <slug> · <term>` | queries_failed | that query dropped |
| All queries failed | `[v0.13] all_queries_failed: <slug>` | rerank_outcome="skipped" | no badge, no nest |
| Rerank LLM call fails after one retry | `[v0.13] rerank_failed_fallback_used: <slug>` | rerank_outcome="fallback" | degraded display (top 1/question by date, "candidate match" phrase) |
| Rerank returns 0 valid picks (abstention) | `[v0.13] rerank_abstained: <slug>` | rerank_outcome="abstained" | no badge, no nest |
| Rerank picks violate INVARIANT | `[v0.13] rerank_invariant: <slug> · violators dropped` | rerank_outcome="picked_N" | partial result |
| Orchestrator throws unexpectedly | `[v0.13] orchestrator_exception: <slug> · <error>` | rerank_outcome="failed" | no badge, no nest; study NOT dropped |

### Minimum-success gate

End of `build:day` summary line:

```
[v0.13] populated: M of K eligible (X%)  ·  skipped: S (no_related_search)  ·  queries fired: P · cache hits: Q
```

Eligible = studies with non-null `related_search`. Skipped = studies the LLM correctly chose not to seek trials for (no clear question to map). WARN banner if `X < 50%` for eligible OR if `S > K` (more skips than picks). Cron stays green; outages and quality degradations both become loud (codex #15).

```bash
# Pin a curator-vetted set (fully validated)
npm run override -- --date=2026-05-20 --related-trials-set=<slug> \
  --json='[{"nct":"NCT05123456","brief_title":"...","overall_status":"RECRUITING","phase":["PHASE3"],"enrollment_count":1200,"primary_completion_date":"2027-03","brief_summary":null,"conditions":["Prostate Cancer"],"interventions":[{"name":"Darolutamide","type":"DRUG"}],"eligibility_brief":null,"answers_question":"Does deeper AR blockade extend OS in nmCRPC?","relevance_phrase":"head-to-head ARSi intensification"}]'
```

`set` validation: NCT format `NCT\d{8}`, status ∈ filter set, phase entries valid, date format `YYYY-MM`, phrase ≤ 60 chars, ≤ 5 entries, NCT uniqueness, every `answers_question` in CURRENT `study.open_questions` (codex #24).

Studio TUI menu entry per study: "Edit watching trials → [suppress all | suppress one question | set exact | clear override]". Available even when `related_trials` is null so curators can pin a set after retrieval failure (codex #26).

Override application order in `applyDigestOverrides()`:

1. Orchestrator writes `related_trials` + `related_trials_provenance`
2. Override layer loads any override for the slug
3. `suppress (no questions)` → `related_trials = []`
4. `suppress (questions=[…])` → filter trials whose `answers_question` is in the suppress list
5. `set` → re-validate against CURRENT `study.open_questions`; drop entries whose `answers_question` no longer matches (WARN log per dropped — codex #23); for remaining RECRUITING entries, look up live CT.gov status and prefer the live value over the pinned status (codex #25)
6. `set` writes also set `related_trials_provenance.rerank_outcome = "pinned_by_curator"`

## Test design

- **Injectable deps:** `buildOneDate(opts)` + orchestrator accept `{ ctgovFetch?, clock?, sleep?, rerankClient?, backfillCache? }`. Tests inject stubs.
- **`backfill-smoke.test.ts:75` mock** extended for the rerank phase; backfillCache threaded across dates.
- **Eval determinism:** 6 fixtures, one per domain (drug, RT, surgery, biomarker, sequencing, methodology) plus a broad-query-failure case (7 total). Each fixture stores recorded CT.gov JSON, indexed by SCENARIO label, not by query hash. The eval stub serves fixtures by scenario the test names; semantically-equivalent prompt changes don't miss fixtures (codex #31).
- **INVARIANT 1 (nct ∈ candidates):** tests cover exact match, fabricated nct, valid nct that wasn't returned this build.
- **INVARIANT 2 (answers_question ∈ study.open_questions):** tests cover exact match, whitespace drift, case drift, fabricated question.
- **Multi-query partial failure:** 0/3, 1/3, 2/3, 3/3 success matrix; rerank runs whenever ≥ 1 succeeded.
- **Deterministic-degraded fallback:** rerank exception → fallback returns ≤ 3 picks (1 per question, sorted by primary_completion_date).
- **Override stale-string drop:** override with `answers_question` not in current open_questions is dropped with WARN.
- **Pinned-status refresh:** override pins RECRUITING; live CT.gov stub returns COMPLETED; merged shows COMPLETED-but-filtered (= dropped).
