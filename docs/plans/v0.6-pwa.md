## Why

The site is already mobile-first (Newsreader serif, warm off-white, 90-second scanning). Curator-readers open it on phones during conference downtime. Two specific gains from going PWA:

1. **Install to home screen.** Removes the URL-bar / tab juggling that's currently between the curator's reader and the digest. One tap from the home screen, full-bleed, no chrome.
2. **Offline-cache the latest digest.** Conference Wi-Fi is hostile. If yesterday's digest is already cached, the reader gets it instantly with no network round-trip.

Push notifications are a separate, **optional** Phase B. Most subspecialist readers won't enable browser push, and email/Substack-style subscriptions are usually a better fit. Push is scoped here so the manifest design doesn't preclude it, not because it's the headline.

## Non-goals

- A real native app (no React Native, no Capacitor).
- Background sync of new digests (the curator publishes once a day; pull-on-open is sufficient).
- Offline editing/admin (admin runs locally; PWA is read-only).
- Push notifications as a v0.6.0 requirement (Phase B, behind a flag, may slip to v0.6.x).

## Scope — Phase A (core PWA)

### Phase A.1 — Web App Manifest

- `public/manifest.webmanifest` — JSON manifest with `name`, `short_name`, `start_url: '/'`, `display: 'standalone'`, `background_color: '#f7f5f0'` (matches body bg), `theme_color: '#f7f5f0'`, `icons[]`.
- `src/layouts/Base.astro:14` (`<head>` block) — add:
  - `<link rel="manifest" href="/manifest.webmanifest" />`
  - `<meta name="theme-color" content="#f7f5f0" />`
  - `<meta name="apple-mobile-web-app-capable" content="yes" />`
  - `<meta name="apple-mobile-web-app-status-bar-style" content="default" />`
  - `<link rel="apple-touch-icon" href="/icons/icon-192.png" />`

### Phase A.2 — Icons

Existing `public/favicon.svg` + `public/favicon.ico` are sufficient for browser tabs but not for installability. Generate PNGs from the SVG once and commit them. Required:

- `public/icons/icon-192.png` (192×192, any-purpose)
- `public/icons/icon-512.png` (512×512, any-purpose)
- `public/icons/icon-512-maskable.png` (512×512, maskable, with safe-zone padding)

Generation script: `scripts/generate-pwa-icons.sh` — uses `rsvg-convert` or `sharp` to rasterize from `public/favicon.svg`. Committed PNGs are the source of truth; script is for regeneration when the SVG changes.

### Phase A.3 — Service worker (offline cache)

- `public/sw.js` — registered from `Base.astro` via inline `<script>` after `DOMContentLoaded`. Scope: `/`.
- **Cache strategy:**
  - **App shell** (`/`, `/about/`, `/sites/`, layout CSS, fonts from google CDN): cache-first with stale-while-revalidate.
  - **Latest digest** (`/<latest-date>/`, its JSON, its images): network-first with fallback to cache.
  - **Older digests**: network-only (don't pre-fill cache; only cache on first visit).
  - **Twitter widget JS** (`platform.twitter.com/widgets.js`): network-only. If offline, source-cards fall back to the existing blockquote — already graceful.
- **Cache versioning:** cache name includes `package.json` version (`oncbrain-v0.6.0`). On activation, old caches are purged. This guarantees clients pick up new asset bundles after each deploy.
- **Build wiring:** the build emits the current version into `sw.js` via an Astro-side template — either a pre-build script that string-replaces a `__VERSION__` token, or Astro's static asset pipeline if it can handle a JS file in `public/` with substitution. Decide at implementation time.

### Phase A.4 — Install affordance

- `src/components/InstallPrompt.astro` — small dismissible footer pill: "Install oncbrain". Listens for `beforeinstallprompt`, stashes the event, shows the pill, calls `event.prompt()` on click. Hidden in standalone mode (`window.matchMedia('(display-mode: standalone)').matches`).
- Mounted in `Base.astro` once, after the disclaimer footer.
- iOS Safari has no `beforeinstallprompt` — leave the pill off on iOS to avoid dead UI. iOS users can still use the share-sheet "Add to Home Screen" manually; we don't surface that prompt.

## Scope — Phase B (push notifications, OPTIONAL)

Behind an env flag (`PUBLIC_PWA_PUSH_ENABLED=true`). Default off for v0.6.0 release. Ship A first, evaluate demand, ship B only if a real reader asks.

If shipped:

- VAPID keypair generated once; public key in `PUBLIC_VAPID_KEY`, private key in the daily-build cron's environment only.
- New `push_subscriptions` SQLite table (endpoint, p256dh, auth, subscribed_at) keyed by endpoint.
- `admin/server.ts` route `POST /push/subscribe` to accept registrations from the PWA.
- `scripts/daily-build.sh` — after successful `git push`, sends a Web Push to every subscription with the new digest's date.
- Unsubscribe flow: deleted subscription on 410 Gone from the push service.

**Risks for Phase B:**
- iOS push requires iOS 16.4+ AND the site be installed via Add to Home Screen. Few subspecialists will jump through that.
- Sending requires a long-lived secret (VAPID private). Storage in launchd env is workable but is one more secret to rotate.
- Web Push delivery is best-effort, not guaranteed.

## Test plan

**Automated:**
- Vitest: cache-name versioning, route-matching logic in `sw.js` (extracted as a testable module if possible). 298+ existing tests still pass.
- `npx astro check`: 0 type errors maintained.
- A manifest schema validator (e.g., `web-app-manifest-validator` npm package, run as a one-off in test).

**Manual:**
- Lighthouse PWA audit on `npm run preview`: ≥90 installability score.
- iOS Safari: Share sheet → Add to Home Screen → open from home screen → confirm standalone display, no URL bar, theme color matches.
- Android Chrome: install banner appears → install → open from launcher → confirm same.
- Offline test: install, load latest digest, enable airplane mode, reopen → digest renders from cache.
- Stale-cache test: deploy a new version, reload the app twice → second reload picks up new content (SW activation behavior).

## Risks & open questions

1. **DO static-site headers.** Service workers need `Content-Type: application/javascript`. DO serves `.js` correctly by default — confirm at implementation time. `Service-Worker-Allowed` header is only needed if SW scope is broader than its path; ours isn't.
2. **Cache invalidation correctness.** A bad SW deploy can lock readers on a broken version. Mitigation: include the activation step that purges old caches, and test the upgrade path manually before shipping.
3. **Google Fonts in offline cache.** The current `<link>` to fonts.googleapis.com is render-blocking. Caching it lets offline renders look right, but adds a cache entry that can go stale across Google's font CDN updates. Plan: cache it with a long-lived stale-while-revalidate window. Acceptable risk; the worst case is a font version drift.
4. **Twitter widget in standalone mode.** `platform.twitter.com/widgets.js` is third-party; in some standalone configs it may behave oddly. The existing blockquote fallback handles this — verify on iOS standalone.
5. **No real "app icon" in the brand yet.** Current favicon.svg is the source. If the curator wants a different mark for the home-screen icon, that's a design pass before Phase A.2 finalizes.
