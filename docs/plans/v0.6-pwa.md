## Why

The site is already mobile-first (Newsreader serif, warm off-white, 90-second scanning). Curator-readers open it on phones during conference downtime. Two specific gains from going PWA:

1. **Install to home screen.** Removes the URL-bar / tab juggling that's currently between the curator's reader and the digest. One tap from the home screen, full-bleed, no chrome.
2. **Offline-cache the latest digest.** Conference Wi-Fi is hostile. If yesterday's digest is already cached, the reader gets it instantly with no network round-trip.

Push notifications are a separate, **optional** Phase B. Most subspecialist readers won't enable browser push, and email/Substack-style subscriptions are usually a better fit. Push is scoped here so the manifest design doesn't preclude it, not because it's the headline.

## Non-goals

- A real native app (no React Native, no Capacitor).
- Background sync of new digests (the curator publishes once a day; pull-on-open is sufficient).
- Offline editing/admin (admin runs locally; PWA is read-only).
- Push notifications as a v0.6.0 requirement (Phase B, behind a flag, may slip to v0.6.x).

### Stack

Use **`@vite-pwa/astro`** (Workbox under the hood) instead of a hand-rolled service
worker. Rationale: Workbox precaches content-hashed assets, so cache invalidation is
automatic and correct ‚Äî the original plan's own #1 risk ("a bad SW deploy can lock
readers on a broken version") is exactly what a hand-rolled `__VERSION__`-token SW
gets wrong and Workbox gets right. Icons via `@vite-pwa/assets-generator`. Both are
dev dependencies.

- **`registerType: 'autoUpdate'`** (silent; `skipWaiting`+`clientsClaim`). No update-prompt UI. Accepted risk: a deploy landing mid-session can leave an open page requesting a dropped hashed chunk; on a content MPA every navigation is a full fresh load, so it self-heals. Documented, not mitigated further.
- **`globPatterns` = shell only** ‚Äî home, `/about`, `/sites` index, CSS/JS, self-hosted font files, icons. NOT dated/study pages (archive grows one page/day forever ‚Äî precaching all of it is unbounded).
- **Latest digest precached at build time** ‚Äî the build knows the latest published date; inject that single `/<latest-date>/` page (and its assets) via `additionalManifestEntries`. This is what makes "latest digest offline" hold for a reader who installs from the home screen but never opened today's page (Codex outside-voice catch #1). Re-pointed each build; bounded to one page.
- **Dated + study pages ‚Üí `NetworkFirst`** with `networkTimeoutSeconds: 3` (don't hang on hostile wifi) and `ExpirationPlugin({ maxEntries: 30 })` (bound the runtime cache).
- **Home `/` + `search-index.json` ‚Üí `StaleWhileRevalidate`** (NOT precache) ‚Äî both change daily; precaching bloats the manifest re-downloaded each deploy and risks a home-freshness regression (installed reader sees yesterday's recent-studies feed for one navigation after a deploy).
- **Twitter `widgets.js` ‚Üí NetworkOnly** (existing blockquote fallback degrades gracefully offline).
- **`navigateFallback: '/offline/'`** with **`navigateFallbackDenylist`** excluding `/api/`, `/feed.xml`, `/slides/`, and asset extensions ‚Äî a broad fallback would otherwise silently serve `/offline/` to feed readers / API consumers (Codex catch #7).

### Phase A.3 ‚Äî Service worker (offline cache)

- `public/sw.js` ‚Äî registered from `Base.astro` via inline `<script>` after `DOMContentLoaded`. Scope: `/`.
- **Cache strategy:**
  - **App shell** (`/`, `/about/`, `/sites/`, layout CSS, fonts from google CDN): cache-first with stale-while-revalidate.
  - **Latest digest** (`/<latest-date>/`, its JSON, its images): network-first with fallback to cache.
  - **Older digests**: network-only (don't pre-fill cache; only cache on first visit).
  - **Twitter widget JS** (`platform.twitter.com/widgets.js`): network-only. If offline, source-cards fall back to the existing blockquote ‚Äî already graceful.
- **Cache versioning:** cache name includes `package.json` version (`oncbrain-v0.6.0`). On activation, old caches are purged. This guarantees clients pick up new asset bundles after each deploy.
- **Build wiring:** the build emits the current version into `sw.js` via an Astro-side template ‚Äî either a pre-build script that string-replaces a `__VERSION__` token, or Astro's static asset pipeline if it can handle a JS file in `public/` with substitution. Decide at implementation time.

### Phase A.4 ‚Äî Install affordance

- `src/components/InstallPrompt.astro` ‚Äî small dismissible footer pill: "Install oncbrain". Listens for `beforeinstallprompt`, stashes the event, shows the pill, calls `event.prompt()` on click. Hidden in standalone mode (`window.matchMedia('(display-mode: standalone)').matches`).
- Mounted in `Base.astro` once, after the disclaimer footer.
- iOS Safari has no `beforeinstallprompt` ‚Äî leave the pill off on iOS to avoid dead UI. iOS users can still use the share-sheet "Add to Home Screen" manually; we don't surface that prompt.

## Risks & open questions

1. **DO static-site headers.** Service workers need `Content-Type: application/javascript`. DO serves `.js` correctly by default ‚Äî confirm at implementation time. `Service-Worker-Allowed` header is only needed if SW scope is broader than its path; ours isn't.
2. **Cache invalidation correctness.** A bad SW deploy can lock readers on a broken version. Mitigation: include the activation step that purges old caches, and test the upgrade path manually before shipping.
3. **Google Fonts in offline cache.** The current `<link>` to fonts.googleapis.com is render-blocking. Caching it lets offline renders look right, but adds a cache entry that can go stale across Google's font CDN updates. Plan: cache it with a long-lived stale-while-revalidate window. Acceptable risk; the worst case is a font version drift.
4. **Twitter widget in standalone mode.** `platform.twitter.com/widgets.js` is third-party; in some standalone configs it may behave oddly. The existing blockquote fallback handles this ‚Äî verify on iOS standalone.
5. **No real "app icon" in the brand yet.** Current favicon.svg is the source. If the curator wants a different mark for the home-screen icon, that's a design pass before Phase A.2 finalizes.
