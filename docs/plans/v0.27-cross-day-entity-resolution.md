# v0.27 — cross-day entity resolution

**Status:** plan (autoplan review, 2026-07-11). Supersedes the entity-resolution
half of `docs/plans/v0.16-entity-resolution-and-source-tagging.md` (that doc
predates v0.26 and chose curator-aliases before the dedup key existed).

## Problem

A trial seen on day N should keep its identity on day N+1 so per-study prior
context resolves across days. Today `loadStudyContext(slug)` (`study-retrieval.ts`,
called at `llm-pipeline.ts:1125` in Phase 2) does an EXACT `data/studies/<slug>.md`
lookup. When `assignSlugsForDate` disambiguates a same-name collision
(`prestige-psma` → `prestige-psma-2`), or a trial is named differently across days
(`PRESTIGE` vs `prestige-psma`), the dossier no longer resolves and the prior
context is silently dropped.

**The trap (do NOT do the naive thing):** stripping a trailing `-\d+` off the slug
to normalize also strips real trial identity — `rtog-0539` / `rtog-0848` →
`rtog`, cross-linking distinct trials. From the SLUG alone you cannot distinguish
a disambiguation suffix (`-2`) from a trial number (`-0539`).

## What v0.26 changed

v0.26 shipped `studyDedupKey(name)` (`src/lib/study-dedup.ts`): a discriminating
identity key derived from the study NAME, not the slug. It already:
- keeps `EORTC 22922` ≠ `EORTC 22033`, `rtog-0539` ≠ `rtog-0848`, `PEACE-2` ≠ `PEACE-V`
- collapses spelling variants (`EORTC-22922` == `EORTC 22922`)
- guards cooperative-group / society prefixes

So the "you can't tell identity from a suffix" premise no longer fully holds: the
dedup key IS a safe cross-day identity signal for name-variant cases — computed
from the name, where the trap doesn't apply.

## Approaches (the premise decision)

- **A. Curator-declared aliases** (the v0.16 design). A dossier opts into slugs via
  `aliases: [prestige-psma, prestige-psma-2, prestige]` YAML frontmatter.
  `loadStudyContext` tries exact, else scans an alias index. Curator-controlled,
  zero false positives, but only resolves studies the curator hand-writes a dossier
  for and manually lists every slug variant — high curator burden, and it does
  nothing until a dossier exists.
- **B. Auto-resolution via `studyDedupKey`.** Build a per-build index `dedupKey →
  [{date, slug}]` from published digests; `loadStudyContext` falls back to "same
  dedup key" when the exact slug misses. Zero curator burden, resolves every
  same-key study automatically, safe against the rtog trap by construction. Risk:
  a wrong merge is silent (feeds the wrong prior context to Phase 2), and the key
  is name-derived so a genuinely different trial sharing a bare name could collide
  (rare, guarded, but not impossible).
- **C. Hybrid (recommended candidate).** Exact slug → curator alias (authoritative
  override) → `studyDedupKey` auto-fallback. Auto-resolution does the work by
  default; a curator alias is the escape hatch to force/suppress a link. NCT, when
  both sides have one, is a hard confirm/deny on the auto path (different NCT ⇒
  never auto-merge, same as the v0.26 detector's conflicting-NCT guard).

## Scope (this plan)

IN: the dossier-context resolution path (`study-retrieval.ts` + a build-time index)
+ tests + a real `build:day` verification. OUT (defer): the rendered-site "trial
history across days" card-merge (bigger, UI); source-tagged Phase 2 claims
(separate v0.16 §2).

## Implementation sketch (pending premise pick)

- `parseDossierAliases(content)` — pure, parses `aliases:` frontmatter, each alias
  `isSafeSlug`-validated.
- Strip frontmatter from the returned dossier body (so the LLM never sees the
  `aliases:` line as content). Existing dossiers have no frontmatter → unaffected.
- Build the resolution index once per build (alias index and/or dedupKey index),
  not a directory scan per study (`loadStudyContext` is per-study).
- Keep the path-traversal guard + `isSafeSlug` on every candidate.
- If auto (B/C): a `dedupKey → newest prior {date, slug}` map from `listDigests()`;
  resolve the incoming study's name → key → prior slug → dossier. Different-NCT
  guard on the auto path.

## Verification

Unit tests for `parseDossierAliases`, the alias/key resolution, frontmatter
stripping, the traversal guard, and (auto path) the different-NCT deny + the
rtog-0539/0848 non-collision. Then a real `build:day` on a date whose study
resolves to a prior dossier under a variant slug, confirming the context reaches
Phase 2.

## Review decision (autoplan, 2026-07-11 — primary + Codex; Claude subagents hit session limit)

**Premise → A (curator aliases), with `studyDedupKey` repurposed to SUGGEST
aliases, never to auto-inject.** Both voices independently landed on A.

- **The C "safety guard" can't run.** `StudyCluster` (llm-pipeline.ts:676) carries
  no NCT, and Phase 2 doesn't know the validated `study.nct` until AFTER the LLM
  call — so the different-NCT deny that made auto-resolution "safe" has nothing to
  deny on at injection time. C silently degrades to B, and B auto-injects on
  exactly the no-NCT preview case that is most ambiguous.
- **Asymmetric risk.** Silent WRONG prior context poisons a clinical analysis
  invisibly (trust-corrupting); silent MISSING context is recoverable. For a
  single-curator clinical tool the asymmetry is decisive.
- **Contract.** `studyDedupKey` is self-described as "a DETECTOR, never an
  auto-merger" (study-dedup.ts:10-17). Auto-injecting prompt context violates it.
- **Synthesis (the value of v0.26 here):** use the dedup-key index to POWER an
  alias-SUGGESTION affordance — `find:dups`-style, "these two cards look like the
  same trial; add an alias?" — so aliases stay authoritative (A) while the curator
  burden of hand-listing variants drops. Suggestion, not injection.

**Implementation (A):**
- `parseDossierAliases(content)` — pure, parses `aliases:` frontmatter, each alias
  `isSafeSlug`-validated; strip frontmatter from the returned dossier body.
- `loadStudyContext`: exact `<slug>.md` → alias index. Build the alias index ONCE
  per build (module memo), never a directory scan per study. Keep the traversal guard.
- (Optional, later) an `npm run suggest:aliases` that prints dedupKey-clustered
  cross-date slugs the curator can paste into a dossier's `aliases:`.

## Scalability (infra check, ~1000 studies)

**The LLM backend is insulated from corpus size — 1000 studies is fine.** Phase 2
issues one LLM call per study IN TODAY'S BUILD (a handful; up to ~20 on a big
meeting day), concurrency 4 (llm-pipeline.ts:753,770). Corpus size adds ZERO LLM
calls to a daily build. This feature adds zero LLM calls; it injects ONE bounded
curator dossier per study, so per-study token cost tracks dossier length, not
corpus size (Codex concurs).

Watch-items (none block 1000 studies, but name them):
1. **Index build cadence** — build the resolution index ONCE per build from the
   `listDigests()` cache (digest-data.ts:391, signature-cached). Never rebuild or
   directory-scan per study: that is current-day-studies × corpus-size, and in a
   `--backfill` it is dates × corpus-size. The whole design turns on this.
2. **`--backfill` over the full corpus** — rebuilds every date = every study
   through the LLM (~1000 Phase 2 calls) and exhausts the claude-cli session
   window (~17 study-agents/window). Pre-existing, not from this feature; chunk
   backfills. Daily builds are unaffected.
3. **Astro static build** — `listStudyPages()` emits one page + one OG PNG (resvg
   render) per study. At 1000 studies that is ~1000 pages + ~1000 OG renders =
   O(n) DEPLOY-time CPU (not LLM), roughly +1-2 min of DigitalOcean build. If it
   bites: cache OG PNGs across builds (regenerate only when the study changes) or
   generate on demand. This is the only genuinely O(n) growth path.
4. **Dossier size discipline** — keep dossiers bounded curator notes. The deferred
   "trial history across days" card-merge must NOT stuff accumulated history into
   the Phase 2 prompt, or per-study tokens grow with a trial's history.
