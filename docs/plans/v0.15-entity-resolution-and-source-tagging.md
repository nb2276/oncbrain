## 1. Cross-day entity resolution

**Goal (TODOS):** a trial keeps its identity across days; `"PRESTIGE"` vs `"prestige-psma"` resolve to the same study so prior-context dossier retrieval (`src/lib/study-retrieval.ts loadStudyContext`) works across days.

### The trap (found this session — do NOT do the naive thing)

The obvious "safe" fix is to strip the per-date disambiguation suffix that `assignSlugsForDate` appends on a same-name collision (`prestige-psma` → `prestige-psma-2`) so the dossier still resolves. **This is unsafe:** the regex `-\d+$` also strips real trial identifiers where the number IS the identity — `rtog-0539`, `rtog-0848`, `nrg-gu005` → `rtog` / `nrg-gu005`→`nrg`. From the slug alone you cannot distinguish a disambiguation suffix (`-2`) from a trial number (`-0539`), so auto-stripping cross-links distinct trials to the wrong dossier. Unacceptable for a clinical product.

### The safe design

**Curator-declared aliases, not auto-stemming.** A dossier (`data/studies/<slug>.md`) opts into the slugs it covers via YAML frontmatter:

```markdown
---
aliases: [prestige-psma, prestige-psma-2, prestige]
---
<the dossier body>
```

`loadStudyContext(slug)`:
1. Exact `<slug>.md` (current behavior).
2. Else scan `data/studies/*.md`, parse each `aliases:` line, return the dossier whose alias list includes `slug`.

Nuances to get right (each is a test):
- **Strip the frontmatter from the returned body** so the LLM doesn't see the `aliases:` line as content (the current code returns the whole file; existing dossiers have no frontmatter so they're unaffected).
- `parseDossierAliases(content)` is pure + testable; each alias must pass `isSafeSlug`.
- Build an alias index once per build, not a directory scan per study (Phase 2 calls `loadStudyContext` per study).
- The path-traversal guard stays.

**Separately**, the rendered-site "this trial's history across days" merge (showing a trial's prior coverage on its card) is a bigger feature that builds on the existing `nct-coverage.ts` index (NCT → dates). Scope it as v0.15.x after the dossier-alias piece.

**Verification:** unit tests for `parseDossierAliases` + the alias resolution; a real `build:day` on a date whose study has a curator dossier under an alias, confirming the context flows into Phase 2.

## 2. Source-tagged Phase 2 claims

**Goal (TODOS, v0.5.1 hardening):** tag each `details` bullet with the source it came from, so mixed tweet + paper + slide + PDF inputs can't silently blend numbers across sources. The numeric validator already cross-checks each table cell against source content; this extends attribution to every bullet.

### Why it can't be shipped "blind"

It is **prompt-dependent**: the Phase 2 study agent (`prompts/digest-v5-study-agent.txt`) must emit a per-bullet source ref, and the only way to confirm the model actually attributes correctly (vs. guessing/blending) is a real `build:day` against multi-source studies and inspecting the output. Shipping a prompt change + a schema field without that verification is exactly the kind of unverified change to avoid in a clinical pipeline.

### Design

- **Schema:** extend `DigestDetail` so a bullet can carry an optional `source: DigestSourceRef` (the typed ref already used by `source_ids`). Back-compat: absent → unattributed (renders as today).
- **Prompt:** instruct the study agent to attach, per bullet, the source id it drew the claim's numbers from; keep it OPTIONAL (don't force a guess — an unattributed bullet is better than a wrong attribution).
- **Validator:** when a bullet cites a numeric claim, cross-check the number against THAT source's content (extends the existing table-cell validator), and drop/flag a bullet whose number isn't in its cited source.
- **Render:** a small source pill per attributed bullet on `StudyCard` (reuse the source-type pill vocabulary).

### Verification (the gate)

Real `build:day` on a study with ≥2 distinct sources carrying different numbers; confirm each bullet's attribution matches the source the number actually came from. Add an eval axis (`build/eval.ts`) for attribution correctness so it's regression-gated.
