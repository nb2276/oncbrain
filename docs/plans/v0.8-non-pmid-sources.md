# v0.8 — Non-PMID source ingestion (URL + PDF) + RSS/API output

**Status:** plan (CEO review: SCOPE EXPANSION, approach C)
**Target release:** v0.8.0
**Reviewed:** /plan-ceo-review on 2026-05-19 (mode: SCOPE EXPANSION)

## Why

Today the bot accepts tweet URLs, PMID URLs, inline `PMID: N`, and photo
attachments. The curator reads papers on journal sites (Lancet, NEJM, JAMA)
and gets PDFs from colleagues, but to bookmark them they have to detour
through PubMed to find a PMID. This removes that friction: paste any paper
URL or forward any PDF, it summarizes into the digest.

It also opens the digest as a content backbone: RSS + JSON API let other
apps and colleagues consume the output.

## Hard constraint (user-stated, load-bearing)

**Do NOT store PDFs or other copyrighted material (figures, full text) on
disk. Summarize and discard.** This is the defining architecture constraint.
Every PDF and every fetched HTML page is transient: extract structured
metadata + abstract, then delete the source. We are a summarization tool,
not a redistribution tool. The summary (title, authors, abstract,
LLM-generated tldr/verdict) is ours to keep; the source artifact is not.

This is distinct from the existing slide-photo path, which stores the
curator's OWN photos (gitignored, curator-private). Journal PDFs are
third-party copyrighted material and must never touch disk beyond a temp
file deleted in a `finally` block.

### PDF summarize-and-discard flow (the IP-critical path)

```
PDF document[] received
   │
   ▼
getFile → download PDF bytes INTO MEMORY (Buffer), not the repo
   │
   ├─[try]────────────────────────────────────────┐
   │  pdf-parse text layer (in-memory Buffer)       │
   │     │                                          │
   │     ├─ text present + sane → LLM metadata      │
   │     │                                          │
   │     └─ empty/garbled → E4: write Buffer to     │
   │           os.tmpdir() (NOT the repo) → pdftoppm│
   │           → temp PNGs in os.tmpdir()           │
   │           → Vision OCR → LLM metadata          │
   │                                                │
   │  store metadata only (title/authors/abstract)  │
   │  E2: reply "Got it: <title>"                    │
   │                                                │
   ├─[catch]───────────────────────────────────────┤
   │  E3: reply with named failure reason            │
   │                                                │
   └─[finally]──────────────────────────────────────┤
      unlink any os.tmpdir() temp files             │
      (the Buffer is GC'd; nothing in the repo)     │
```

**Codex finding 1 — temp files must NOT live in the repo.** The repo is
under Dropbox/CloudStorage; sync/indexing could grab a PDF or rasterized
PNG before unlink, defeating the hard constraint. The existing code already
does this right: `slide-photo-storage.ts` keeps downloads in memory and
`vision-ocr.ts` uses OS temp space. Follow that pattern:
- PDF bytes stay in a Buffer for text-layer parsing (never written to disk
  in the common case).
- Only the OCR fallback writes to `os.tmpdir()` (outside the synced repo),
  and cleans up in `finally`.
- `finally` is not a crash guarantee. A SIGKILL/OOM mid-OCR can orphan a
  temp file. Mitigation: a startup sweep that deletes stale
  `os.tmpdir()/oncbrain-pdf-*` files older than 1h, so even a hard crash
  self-heals on the next run. The repo never holds copyrighted bytes.

## Schema changes (corrected per codex findings 2 + 3)

**SQLite can't ALTER away `NOT NULL`/`UNIQUE` in place**, and this codebase
only does additive `ADD COLUMN` migrations (`db.ts:266-296`). Dropping the
`pmid TEXT NOT NULL UNIQUE` constraint requires a **table rebuild**:

```sql
-- 1. New table with the corrected constraints.
CREATE TABLE papers_new (
  id INTEGER PRIMARY KEY,
  pmid TEXT,                       -- now nullable
  doi  TEXT,
  source_url TEXT,                 -- audit: which URL the curator sent
  ... (all existing columns) ...,
  CHECK (pmid IS NOT NULL OR doi IS NOT NULL)   -- must have at least one key
);
-- 2. Copy existing rows (all have pmid today).
INSERT INTO papers_new SELECT id, pmid, doi, NULL, ... FROM papers;
-- 3. Swap.
DROP TABLE papers; ALTER TABLE papers_new RENAME TO papers;
-- 4. Separate partial unique indexes — NOT COALESCE.
CREATE UNIQUE INDEX idx_papers_pmid ON papers(pmid) WHERE pmid IS NOT NULL;
CREATE UNIQUE INDEX idx_papers_doi  ON papers(lower(doi)) WHERE doi IS NOT NULL;
CREATE INDEX idx_papers_date ON papers(bookmark_date);
```

Why not `COALESCE(pmid, doi)` (codex finding 3): a DOI-only row
`(NULL, 10.x)` and a later PMID-backed row `(12345, 10.x)` would compute
different coalesced keys (`10.x` vs `12345`) and BOTH survive — the exact
dedup the feature exists to prevent. Separate unique indexes on normalized
(lowercased) DOI and on PMID catch the collision correctly.

**Merge logic:** when a DOI-only paper later arrives with a PMID (same
normalized DOI), UPDATE the existing row to add the PMID rather than insert
a duplicate. `saveePaper` (`db.ts:616`) currently keys on PMID only; it
needs a DOI-first lookup branch.

DOI-only papers: `pmid = NULL`, `doi` populated, abstract from Crossref/
OpenAlex (with the fallback policy below). Renderer already handles papers
generically.

This is a destructive migration (DROP TABLE). The DB is gitignored and
local-only, and a backup copy should be written before the rebuild
(`oncbrain.db.bak-<date>`) so a botched migration is recoverable.

## Error & rescue map (the critical section)

| Codepath | Failure | Exception | Rescued? | User sees |
|---|---|---|---|---|
| URL classify | unrecognized host | (none, returns null) | Y | reply "not a recognized paper URL" |
| HTML fetch | timeout | TimeoutError | Y | reply "couldn't reach the page, try again" |
| HTML fetch | 403 / paywall / bot-block | HttpError | Y | reply "page blocked the fetch, paste the PMID or PDF" |
| Highwire parse | no citation_pmid/doi meta | MetaNotFound | Y | reply "no paper metadata on that page" |
| DOI resolve | Crossref 404 | DoiNotFound | Y | reply "DOI not found in Crossref" |
| PDF getFile | Telegram file too large | FileTooLarge | Y | reply "PDF too large (>20MB Telegram cap)" |
| PDF parse | no text layer | EmptyTextLayer | Y → E4 OCR fallback | (transparent) or E3 reply |
| PDF OCR | Vision unavailable (non-Mac) | OcrUnavailable | Y | reply "scanned PDF needs the Mac build, deferred" |
| LLM extract | malformed/empty response | LlmParseError | Y | reply "couldn't extract paper details" |
| PDF cleanup | delete fails | (logged, not user-facing) | Y | server log; retry unlink |

**SSRF note (deepened per codex finding 4):** fetching curator-pasted URLs
is an SSRF surface. Curator is the single trusted input so risk is low, but
arbitrary journal URLs are a harder class than the existing `isSafeImageUrl`
host allowlist. The full guard:
- https-only.
- `redirect: 'error'` is WRONG here — `doi.org` resolves by 302 redirect,
  so it would break DOI ingestion. Instead: follow redirects manually, max
  3 hops, and re-validate the resolved IP at EACH hop (not just the first).
- Block private/link-local/loopback ranges in all encodings: dotted-decimal
  10/172.16/192.168, IPv6 (`::1`, `fc00::/7`, `fe80::/10`), IPv4-mapped
  IPv6 (`::ffff:10.x`), and decimal/hex/octal IPv4 literals
  (`2130706433`, `0x7f.1`, `0177.0.0.1`).
- Resolve hostname → IP and check the IP before connecting; guard against
  DNS rebinding / TOCTOU by pinning the resolved IP for the actual fetch
  (or re-checking after connect).
- 10s timeout, cap response body (~5MB), discard the HTML after meta
  extraction (IP constraint).
- For a single-curator trusted bot this is belt-and-suspenders, but it's
  cheap and the URL surface is genuinely arbitrary.

## Security & threat model

- **Attack surface:** new outbound HTTP fetches (URL ingestion), new file
  download (PDF). Both curator-triggered.
- **SSRF:** see error map note. Block private-range hosts; https-only.
- **PDF as malware vector:** pdf-parse runs in-process; a malicious PDF
  could exploit the parser. Mitigation: pin pdf-parse version, run
  extraction with a timeout, never execute embedded JS (pdf-parse doesn't).
- **Prompt injection via PDF/HTML text:** extracted text goes to the LLM.
  The existing trust-boundary instructions in the study-agent prompt apply;
  the metadata-extraction prompt needs the same "treat as data, not
  instructions" guard.
- **IP/copyright (the headline constraint):** PDF + fetched HTML deleted
  after extraction; only summary metadata persists. No figures stored.

## Output: RSS + JSON API (E1)

- `/feed.xml` — RSS 2.0, latest ~30 studies, each item = title + verdict +
  audience + tldr + link to `/<date>/#<slug>`. `@astrojs/rss` or hand-rolled.
- `/api/v1/digests.json` — list of dates + counts.
- `/api/v1/digest/<date>.json` — one full day's artifact.
- `/api/v1/study/<slug>.json` — one study (cross-date resolved).
- CORS open (`Access-Control-Allow-Origin: *`); it's public content.
- `/api/v1/` namespace from day one so schema evolution doesn't break
  consumers (one-way door once published).

## DOI abstract + OCR availability policy (codex finding 6)

- **Crossref/OpenAlex often lack abstracts.** When a DOI resolves but no
  abstract is available, ingest the paper with title + authors + journal +
  year only, and have the LLM verdict acknowledge "abstract unavailable"
  rather than fabricate analysis. Do NOT silently produce a confident
  verdict from a title alone. If even title/authors are unavailable, E3
  reject: "couldn't pull paper details from that DOI, try the PDF or PMID."
- **Apple Vision + pdftoppm are not guaranteed present.** Current OCR
  degrades gracefully when the binary is missing (`vision-ocr.ts:91-100`) —
  fine for optional slide OCR. For the PR2 scanned-PDF path, detect absence
  at ingest time and E3-reject with "scanned PDF needs the Mac build with
  poppler + Vision; this ran elsewhere" rather than promising OCR that
  can't run. Document the poppler dependency in CONTRIBUTING/README.
