# v0.6 — Live search

**Status:** plan
**Target release:** v0.6.0 (can ship independently of the PWA work)
**Source TODO:** [TODOS.md](../../TODOS.md) → v0.6 section
**Time estimate:** ~3 hours for the MVP

## Why

The curator scans across days. A subspecialist looking for "PEACE-3" or "what did they say about HER2-low last week" should find it in one box, not by clicking through each date. As the archive grows past ~10 days, "browse by date" stops scaling.

## Non-goals

- A backend search service. Static site, no server-side index.
- Cross-site full-text search (Google does that fine). This is *inside oncbrain* only.
- Author search, journal search, year filter. Per-study + per-site is enough for MVP.
- Re-ranking by recency, popularity, or relevance score. Match-or-don't is fine to start.
- Source-type filter (tweet/paper/slide). Stretch goal.

## Approach

**Client-side substring filter over a pre-built JSON index.** Lean MVP, no library dependency.

Decision: substring filter over a real client-side search engine (lunr, flexsearch). Reasoning:
- The vocabulary is bounded: trial acronyms (PEACE-3, EXTEND), disease-site labels, biomarker names (HER2-low, PSMA), drug names. Substring matching catches all of these without library overhead.
- "PRESTIG" → "PRESTIGE-PSMA" works via `.includes()` — no fuzzy needed.
- Typo tolerance is the only thing we'd gain from lunr/flexsearch. For a subspecialist audience that types trial names they know, typos are rare. Defer until a real reader complaint.
- Bundle cost: substring filter is ~30 lines of vanilla JS. lunr is ~9KB gz; flexsearch is ~7KB gz. The bundle savings is real for a mobile-first site.

If the simple version proves insufficient later (e.g., archive grows past 500 studies, or typo complaints come in), swap the engine without changing the index emit. The index shape is engine-agnostic.

## Index shape

One entry per study, across all published dates.

```ts
type SearchEntry = {
  date: string;           // "2026-05-18"
  site: string;           // disease-site slug, e.g. "prostate"
  site_label: string;     // "Prostate"
  site_emoji: string;     // "🌰"
  name: string;           // "EXTEND Trial"
  slug: string;           // stable slug for the on-page anchor
  tldr: string;           // study tldr (one sentence)
  nct: string | null;     // "NCT04567890" or null
};
```

URL semantics: `/${date}/#${slug}` deep-links to the study card on the date page.

**Index size:** ~250 bytes/entry × ~10 studies/day × 30 days = ~75KB raw. JSON compresses well; over the wire it's ~15-25KB. Acceptable single fetch for the home page.

**Stability of `slug`:** The Phase 1 grouping prompt already emits a `slug` field per cluster (see `llm-pipeline.ts:465-479`), but that slug is not currently persisted on the final `DigestStudy` in the artifact JSON. Two options:

- **A (recommended):** Persist the slug on `DigestStudy` going forward. New artifacts get a `slug` field; for old artifacts (already published), derive a fallback slug at index-build time via `name.toLowerCase().replace(/[^a-z0-9]+/g, '-')`. Single small schema additive change. Future-proof for any other "link to a specific study" need.
- **B:** Derive client-side from `name` at index-build time and at page-render time. No schema change. Risk: derivation drift if the two code paths ever diverge.

Go with A — it's a strictly additive field, doesn't break v0.5.0 readers, and prevents future drift.

## UI

**Location:** single search input on the home page (`src/pages/index.astro`), above the recent-dates strip. Not in `Base.astro` for the MVP — most navigation already happens from home, and adding a search box to every page costs vertical real estate on mobile.

**Layout (mobile-first):**

```
┌──────────────────────────────────────┐
│ 🔍 Search studies, trials, NCT...    │  ← input, 16px font (iOS no-zoom)
└──────────────────────────────────────┘

(when query non-empty:)
N results

🌰 Prostate · 2026-05-18
EXTEND Trial
EXTEND phase II confirms durable PFS benefit...
→

🫘 Kidney · 2026-05-17
PRESTIGE-PSMA
Lu-PSMA mPFS 14.2 vs 9.8mo HR 0.62...
→
```

**Empty state (no query):** the search input is the only thing shown above the existing recent-dates strip. Don't pre-show results.

**No-results state:** "No studies match 'foo'. Try a trial name, NCT, or disease site."

**Match rules:**
- Case-insensitive substring match.
- Token sources, in order tested: `nct`, `name`, `tldr`, `site_label`.
- Show up to 20 results. If more match, show "+N more" so the reader knows to narrow.
- Sort: most recent date first.

**Keyboard:** `/` focuses the input from anywhere on the page. `Esc` clears + blurs. No autocomplete, no arrow-key navigation in results (stretch goal).

## Build-time index emit

New Astro endpoint: `src/pages/search-index.json.ts`. Iterates all digest artifacts, expands each site's studies into `SearchEntry` rows, emits a single JSON array sorted by date desc.

```ts
import type { APIRoute } from 'astro';
import { listDigests } from '../lib/digest-data.ts';
import { getDiseaseSite } from '../lib/disease-sites.ts';

export const GET: APIRoute = () => {
  const entries: SearchEntry[] = [];
  for (const digest of listDigests()) {
    for (const site of digest.sites) {
      const meta = getDiseaseSite(site.disease_site);
      for (const study of site.studies) {
        entries.push({
          date: digest.date,
          site: meta.slug,
          site_label: meta.label,
          site_emoji: meta.emoji,
          name: study.name,
          slug: study.slug ?? deriveSlug(study.name),
          tldr: study.tldr,
          nct: study.nct,
        });
      }
    }
  }
  return new Response(JSON.stringify(entries), {
    headers: { 'Content-Type': 'application/json' },
  });
};
```

Output: `dist/search-index.json` served at `/search-index.json`.

## Risks and open questions

1. **Index size at archive maturity.** At 200 days × 10 studies/day, the index is ~500KB raw / ~100KB over the wire. Still acceptable for a single fetch (smaller than most image assets), but worth re-measuring at that point. If it gets bigger, switch to date-windowed loading (last 60 days by default, "load more" for older).
2. **Slug collisions across days.** Two studies on different dates can share a slug (`extend-trial` 2026-05-18 + `extend-trial` 2026-05-25). That's fine because slug is scoped to its date page. The search index keys on `(date, slug)` together, never on slug alone.
3. **Slug churn for old artifacts.** Existing artifacts won't have a persisted slug. The index emitter falls back to `deriveSlug(name)`. As long as `deriveSlug` is deterministic, old links keep working. If we ever change the derivation rule, old anchors break — keep the function stable, or add migration.
4. **Search affecting Twitter widget hydration.** The search box is above the recent-dates strip but doesn't interact with the Twitter widget. No expected conflict.
5. **Lazy vs. eager index fetch.** Lazy (fetch on first keystroke) saves bandwidth for the 95% of visits that don't search. Eager (fetch on `DOMContentLoaded`) makes the first keystroke instant. Lean lazy with a small prefetch on `focus` to bridge the gap.
