# v0.24 — OCR of PMC Open-Access figure images

**Status:** SHIPPED in v0.24 (both tiers). See CHANGELOG.
**Motivates:** RTOG 1005 (JCO, PMID 42114027) ingested via PMID has `figure_ocr_md = null` — the pipeline never saw its figures. (Confirmed it is NOT in the OA subset, so it still needs a forwarded PDF; the OA path is for future OA papers.)

## Problem

Figure OCR (the numbers printed *inside* KM curves, forest plots, image-rendered
tables) runs in exactly one place: `enrichPdfPaper` on a forwarded **PDF file**
(`inbox-enrichment.ts:457,461` → `extractPdfFigureOcr` / `extractPdfFigureStructured`).

A paper ingested via a **PMID / DOI / PMC link** gets **text only** — abstract +
Methods/Results excerpt (`pubmed-client.ts` `fetchPubMedPaper`). It never
downloads or OCRs the figure *images*, so any magnitude locked in a figure is
invisible to the Phase 2 study agent. Confirmed: `pubmed-client.ts` and
`oa-enrichment.ts` have zero figure/image handling; `resolveFromPubMed/Doi/Url`
never set the figure fields.

For **open-access** articles, PMC exposes the figure images for free. We can fetch
and OCR them through the *existing* pipeline, getting figures with no forwarded PDF.

## Coverage (honest boundary)

- Helps **only PMC OA-subset** articles. Many oncology trial reports (JCO, Lancet
  Onc, NEJM) are paywalled / not in the OA subset — those still need the PDF.
  This is **additive**, not a replacement for the PDF path. (RTOG 1005 is JCO →
  it will NOT qualify; this is for future OA papers.)
- macOS-only (Apple Vision), same as the PDF path. Runs at local enrichment; the
  DigitalOcean build never touches it.

## Reuse (what already exists)

Nearly all the machinery is image-input-ready:

| Need | Existing function | File |
|---|---|---|
| Raw Vision OCR of an image | `ocrFile(path)` | `vision-ocr.ts:157` |
| Grounded structured extraction of an image | `extractFigure(imagePath, opts)` | `figure-extract.ts:217` |
| Grounding gate (withhold un-OCR'd numbers) | `auditReconciledOutput` / `groundNumbersAgainstOcr` | `figure-extract.ts` |
| Qwen availability / kill switch | `isQwenAvailable`, `FIGURE_STRUCTURED=off` | `qwen-client.ts` |
| macOS OCR gate | `isOcrAvailable()` | `vision-ocr.ts:79` |
| PMCID resolution + PMC XML fetch + rps throttle | `fetchPubMedPaper` | `pubmed-client.ts:93` |

`extractFigure` already takes an **image path** (the PDF path only differs in that
it rasterizes pages to PNGs first via `rasterizePageToPngs`). PMC hands us the
figure images directly, so we skip rasterization and feed them straight in.
**The only new logic is acquiring the images.**

## Two tiers

### Tier A — figure CAPTIONS from the nxml we already fetch (cheap, text-only, cross-platform)

The PMC full-text nxml is **already fetched** for Methods/Results
(`fetchPubMedPaper`, `efetch db=pmc`). Its `<fig><caption>` text often carries the
very numbers we want (authors put KM medians / HRs in captions). Today
`extractMethodsAndResults` filters to Methods+Results and likely drops captions.
Add a `<fig><caption>` extractor and feed captions to the study agent (append to
`fulltext_excerpt_md`, or a new local-only `figure_captions_md`).

- No new fetch, no OCR, no macOS dep, no OA restriction (works for any PMC full text).
- Effort: ~2h. High yield for the cost.

### Tier B — figure IMAGE OCR from the PMC OA package (the ask)

Acquire the OA figure images and run them through the existing OCR/structuring.

**New module `src/lib/pmc-oa.ts`:**
1. **OA gate + package locator:** GET
   `https://www.ncbi.nlm.nih.gov/pmc/utils/oa/oa.fcgi?id=PMC…`.
   Response is either `<link format="tgz" href="…PMC….tar.gz">` (in OA subset) or
   `<error code="idIsNotOpenAccess"/>` (skip). This is BOTH the OA-membership
   check AND the image source in one call.
2. **Download + unpack:** fetch the `.tar.gz` (SSRF-safe / NCBI-host-restricted;
   size cap ~50MB), untar via shelling to the system `tar` (no npm dep — matches
   the poppler "binary, not a dep" ethos) into a temp dir cleaned in `finally`.
3. **Select figures + captions:** the package contains the nxml + all figure image
   files. Parse `<fig>` → `<graphic xlink:href="figN.jpg">` to pick the REAL
   figure images (authoritative — better than the PDF path's raster-size
   heuristic) and pair each with its caption.

**OCR (reuse, ~zero new logic) — `enrichPmcOaFigures(pmcId)`:**
- Each figure image → `ocrFile(img)` → concat into `figure_ocr_md` (same
  `[Fig N]\n…` shape as the PDF path's `[p.N]\n…`).
- If Qwen up → `extractFigure(img)` → grounded structured md → concat into
  `figure_structured_md`. Same v0.20 grounding gate: a number not in the image's
  own Vision OCR is withheld.

**Wire-in (`enrichPaperItem`):** after `resolveFromPubMed/Url` builds the text
`NewPaper`, if `pmc_id` present AND `isOcrAvailable()` AND figures still empty →
call `enrichPmcOaFigures` (best-effort, non-fatal, like the PDF figure step) and
set the fields on `saveInput` before `savePaper`. Kill switch `PMC_OA_FIGURES=off`.

### v0.23 synergy

The richness-aware merge means order doesn't matter: a later forwarded PDF still
UPGRADES over PMC-OA figures if it carries more (fill-if-missing / clean-PDF
upgrade), and a PMC-OA re-ingest that gains figures queues a rebuild — both
already handled by v0.23. No conflict.

## IP / publish boundary

`figure_ocr_md` / `figure_structured_md` are **local-only** (guarded by
`publish-boundary.test.ts` FORBIDDEN_KEYS). Tier B writes the same columns → no
boundary change, still guarded. PMC OA content is openly licensed, but we keep the
same local-only posture (some OA licenses are non-commercial; the site publishes
summaries, not source figures). Tier A captions: feed to the study agent as input;
do not publish verbatim (conservative).

## Failure modes / safety

- Non-OA (`idIsNotOpenAccess`) → skip silently (no figures, exactly as today).
- oa.fcgi / tarball fetch failure → non-fatal (paper ships with text).
- SSRF: restrict fetches to `*.ncbi.nlm.nih.gov` / `ftp.ncbi.nlm.nih.gov`; cap
  tarball size; temp-dir cleanup in `finally`.
- NCBI politeness: reuse the `pubmed-client` rps throttle; send a User-Agent +
  optional `api_key`.
- Non-macOS (`isOcrAvailable()` false) → skip.
- `tar` binary missing → skip (typed, like a missing poppler binary).

## Backfill

`build/backfill-pmc-oa-figures.ts` (mirror `backfill-figure-ocr.ts`): for every
paper with a `pmc_id` and null `figure_ocr_md`, attempt PMC-OA figure OCR + merge
(reuses `savePaper`'s v0.23 fill-if-missing). Retro-fills the OA back-catalog.

## Testing

- `pmc-oa.ts`: mock oa.fcgi (OA + non-OA), a fixture `.tar.gz` (nxml + one figure
  PNG); assert figure selection + caption pairing; host/size guards; untar-missing
  path.
- enrichment: pmc_id + OA → figures set; non-OA → null; OCR-unavailable → null.
- publish-boundary: fields stay guarded (already covered — same columns).

## Effort

- Tier A: ~2h (nxml caption parse + wire-in + test).
- Tier B: `pmc-oa.ts` (~200 lines) + `enrichPmcOaFigures` wire-in (~30) + backfill
  (~60) + tests. Reuses `extractFigure`/`ocrFile`/grounding entirely.
- Total ~1–1.5 days human / a few hours CC.

## Decisions to make

1. **Tiers:** A only / B only / both. (Recommend **both** — A is a cheap high-yield add.)
2. **Acquisition:** OA package tarball (one fetch, authoritative figure list +
   captions, clean OA gate) vs per-figure image URLs. (Recommend **tarball**.)
3. **Untar:** shell `tar` (no dep) vs an npm tar library. (Recommend **shell**.)
4. **Coverage expectation:** accept that this helps only the OA minority; the PDF
   path stays primary for paywalled papers. (Recommend **yes** — additive.)
