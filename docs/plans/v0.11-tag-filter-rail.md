# v0.11 — Tag filter rail (cross-page client-side AND-filter)

Status: DRAFT — pre-CEO/Eng/Adversarial/Codex review.

## Context

v0.10 (just shipped) added a tag system on top of disease-site nav:
typed fields on every study (modality / intent / methodology), per-tag
landing pages at `/tags/<slug>/`, 2-way and 3-way intersection URLs,
per-tag RSS + JSON API, StudyCard chip rows + sibling footer, a home-
page tag activity histogram with sparklines, and a curator override
CLI. Backfill regenerated the existing 9 digests so 41 studies are
tagged at launch.

Two reader-facing gaps surfaced immediately after ship:
1. The activity histogram + tag chip row live BELOW the recent-studies
   feed on the home page — readers have to scroll past 41 study cards
   to see them. Discoverability is effectively zero.
2. The original CEO ask was "add to bottom of left nav." Design review
   pushed the surface to home-page sections instead. The reader feedback
   is unambiguous: they want tags in the nav.

The curator also asked for filter semantics, not just navigation: pick
multiple tags, narrow the visible studies on the current page. This is
a different product from the landing pages — landing pages are
bookmarkable / RSS-able / indexable; the filter is ephemeral,
exploratory, fast, no URL change (unless we add one).

v0.11 ships a left-rail filter on every page that displays study cards.

## Goals

- Add a persistent left-rail TAGS section on the home page, per-site
  pages (`/sites/<site>/`), per-date pages (`/<date>/`), and per-tag
  landing pages (`/tags/<slug>/`).
- Filter semantics: AND (multi-select narrows the visible list).
- URL sync so a filter state is shareable and survives reload:
  `?tags=radiation+phase-3-rct` (alphabetical canonical, matches the
  intersection-landing-page URL form for consistency).
- Filter state applies to the current page only (no cross-page
  persistence). Each page renders its full unfiltered list server-side;
  the filter is a JS hydration layer that toggles `[hidden]`.
- Graceful degrade: with JS disabled or while hydrating, the full
  unfiltered list renders. PWA-cached pages keep working unchanged.
- "Clear all" affordance + active-count badge.
- Accessibility: real `<input type=checkbox>` elements (or
  role=checkbox), keyboard navigation, aria-live region announcing the
  filtered count change.

## Non-goals

- Cross-page filter persistence (e.g. filter set on home doesn't follow
  the reader to a date page). Deferred to v0.11.x if requested.
- Server-side filter rendering. SSR delivers the full list; filter is
  client-side only. The existing landing pages already serve the
  "server-rendered, indexable, shareable" niche.
- OR semantics or NOT semantics. AND only for v1.
- Free-text tag search. The tag set is small (16 slugs across 5
  namespaces); a search box is wrong for this surface.
- Touching the existing TriageRail (intra-page jump list on `/<date>/`
  at ≥1200px). The filter rail and TriageRail must coexist; see Risks.

## UX shape

Left rail, fixed-width column on viewports ≥1024px (matching the
existing SITES nav breakpoint), with a mobile drawer below that.

Section structure:

```

  Bladder
  Breast
  ...

TAGS · 17 active
  ─ Modality ─
  ☐ Radiation (37)
  ☐ Surgery (1)
  ☐ Systemic (2)
  ☐ Combined (3)

  ─ Intent ─
  ☐ Curative (37)
  ☐ Palliative (4)

  ─ Methodology ─

  ☐ Phase 2 trial (8)
  ☐ Meta-analysis (1)
  ☐ Retrospective (3)
  ☐ Real-world evidence (3)

  ☐ Challenges SOC (7)
```

Counts are scoped to the current page: on `/sites/breast/` the counts
show how many breast studies carry each tag (and tags with zero matches
on that page are hidden, since they'd produce an empty result).

Active filter chips render at the top of the study list with a
"Filtering: Radiation + Phase 3 RCT (3 studies) — clear all" header so
the reader sees the filter state without having to scan the rail.

## Architecture

### Tag data on each study card

Server-side: each study card's root `<article>` gets a
`data-tags="modality:radiation intent:curative methodology:phase-3-rct verdict:confirmatory meeting:asco-2026"`
attribute (space-separated namespace:slug tokens). The StudyCard
component already derives these via `deriveStudyTags`; we just emit
them as a data attribute. No extra render cost.

### Filter state

Client-side: a singleton `FilterStore` (vanilla JS, no framework)
holds:
- `Set<string>` of active filter tokens (e.g. `"modality:radiation"`)
- `URLSearchParams`-backed sync (`?tags=radiation+phase-3-rct`)

On hydrate:
1. Parse `?tags=` from `window.location.search`
2. Resolve each slug to a namespaced token using the tag-index
   (slug-only URLs require a lookup table; we ship a small static
   `data-slug-namespace` JSON inlined in the page)
3. Tick the matching checkboxes
4. Apply the filter

On checkbox toggle:
1. Update the Set
2. Push the URL via `history.replaceState` (not pushState — we don't
   want every chip click to be a back-button entry)
3. Re-apply the filter
4. Update the active-chips header + aria-live region

### Filter application

For each study card (`article[data-tags]`):
- Parse `data-tags` into a Set
- If `activeFilterSet.isSubsetOf(cardTagSet)` → show; else `[hidden]`
- Also update per-tag counts ("Radiation (37)" → "Radiation (3)" after
  filtering by Phase 3 RCT)

Performance: 41 study cards × ~5 tags each = 205 tokens to scan per
toggle. Sub-millisecond.

### URL sync edge cases

- `?tags=invalid-slug` → ignore the invalid token, render whatever
  matches the rest. Don't error.
- `?tags=radiation+radiation` (duplicate) → dedup to one.
- 4+ tag intersection → allowed (the landing pages cap at 3-way; the
  in-page filter has no cap because there's no URL pre-generation).
- Slug collision across namespaces would break the lookup, but we
  already assert global uniqueness at build time
  (`assertSlugUniqueness`) — that invariant carries over.

### Suggest canonical landing URL

When the filter state matches a built landing page (single tag, or 2-
3-way intersection that has a generated `/tags/<a>+<b>(+<c>)/` URL),
show a subtle "View at /tags/radiation+phase-3-rct/ →" link in the
active-chips header. Encourages bookmarking the canonical form for
multi-page narrowing; doesn't force the navigation.

### Per-page integration

**Home (`src/pages/index.astro`):** Left rail with SITES (top) + TAGS
(below). Filter narrows the "Studies · 41" recent-studies feed.

**Site (`src/pages/sites/[site].astro`):** Left rail with disease-site
breadcrumb + TAGS. Filter narrows the studies on that site.

**Date (`src/pages/[date].astro`):** Left rail with TAGS only (no
SITES — the page is intra-day, not cross-corpus). Conflicts with
TriageRail at ≥1200px (see Risks). Filter narrows that day's studies.

**Tag landing (`src/pages/tags/[...slug].astro`):** Left rail with TAGS,
with the current tag(s) pre-ticked. Filter further narrows the studies
that carry the URL's tag (so a reader on `/tags/radiation/` can add
"+phase-3-rct" via checkbox without re-typing).

## Mobile behavior

Viewports <1024px: the rail collapses into a button at the top of the
page ("Filters · 2 active"), opening a bottom-sheet drawer with the
same checkbox tree. Drawer is dismissible by tap-outside or close
button. Drawer state is independent from the desktop rail (no shared
scroll position).

Active-chips header renders inline on mobile too (above the study
list), so the reader sees the filter state without opening the drawer.

## Risks + open questions

1. **TriageRail conflict on `/<date>/` pages (≥1200px).** Both want the
   left gutter. Options:
   - A: TAGS rail on left, TriageRail moves to right gutter.
   - B: TAGS rail on left, TriageRail collapses to a top-of-page chip.
   - C: Stack both in left rail (TAGS top, TriageRail bottom). May
     compete for vertical space.
   - **Recommendation:** A. TriageRail is intra-page (study jumps);
     TAGS is cross-corpus. The reader's left-eye scan should be
     dominated by the cross-corpus surface.

2. **Cached PWA pages render the full list pre-hydration.** Acceptable
   — the existing PWA shell shows full content; filter narrows on
   hydration. A reader on a slow connection sees all studies, then the
   filter "pops in." Alternative: lazy-render the JS bundle for the
   filter so it doesn't compete with the article hydrate. Defer.

3. **Empty-state copy.** A filter that matches zero studies shows
   "No studies match this filter. Clear filters to reset." Don't auto-
   clear; that would surprise the reader.

4. **Tag landing page becomes ambiguous.** A reader on
   `/tags/radiation/` who unchecks Radiation in the rail is now seeing
   the full corpus on a URL that says they're filtering to one tag.
   Two options:
   - A: Disable the URL's tag in the checkbox (greyed out, locked
     active).
   - B: Allow unchecking, but show a banner: "You're on
     /tags/radiation/. Showing all studies (filter removed). Return
     to /."
   - **Recommendation:** B. Honors the reader's intent; doesn't trap
     them on a URL.

5. **SEO impact.** The filter is client-side; search engines see the
   full list, not the filtered subset. Landing pages still own SEO.
   No regression.

6. **PWA cache invalidation when adding data-tags.** The data-tag
   attribute change shifts every page's HTML hash, so the next site
   build will trigger a precache update for every cached route. Plan:
   land the data-tag emission as a separate PR before the JS so the
   PWA churn happens once, with no behavior change.

7. **Filter state in URL clutters analytics.** Cloudflare Analytics
   counts `/` and `/?tags=radiation` as distinct paths. Either:
   - A: Strip `?tags=` server-side for analytics.
   - B: Accept the noise.
   - **Recommendation:** B. Filter usage is signal worth keeping.

8. **Slug-namespace lookup table.** The filter needs to know that the
   slug `radiation` is in the `modality` namespace. The build-time tag
   index already computes this; we ship it as a small inline JSON
   blob (`<script type="application/json" id="tag-namespaces">`) on
   each page. ~500 bytes uncompressed; trivial.

## Adversarial + Codex review findings (to amend before PR-1)

Both reviewers ran on the draft plan above. Triage:

### P0 (must amend before any PR ships)

1. **Query encoding broken.** `URLSearchParams` ↔ `?tag=a+b` ambiguity:
   `+` decodes as space; the encoded form would be `%2B`. **Decision:**
   use repeated params `?tag=a&tag=b` (no codec ambiguity, supports
   arbitrary slugs). Plan-wide URL form change.
2. **TriageRail breakpoint mistake.** Real breakpoint is ≥1640px on
   `/<date>/` pages (`<TriageRail wide={true}/>`), not ≥1200px. Risk #1
   collision analysis solves a non-problem. **Decision:** rewrite Risk
   #1 + PR-4 against the real breakpoint; iPad portrait gets the rail
   to itself.
3. **Tag-landing-page uncheck impossibility.** `/tags/<slug>/`
   statically renders only the studies that carry the URL's tag;
   unchecking can't reveal cards that aren't in the DOM. **Decision:**
   resolved by the CEO canonical-URL strategy below — unchecking
   pushes the canonical URL to a different one, not stays on a lying
   URL.
4. **PWA SW rejects non-empty queries.** `pwa-routes.ts:39` archive
   routing rejects all `?*` URLs; cached date/site/tag pages can't
   serve `?tag=` offline. Home is unguarded → query-variant cache
   entries. **Decision:** SW route matchers normalize `tag` params
   OUT of the cache key before lookup (treat `?tag=…` as the same
   cache entry as the bare URL); add to PR-1 (foundational layer).
5. **DOM contract mismatch.** Home renders `<li class="recent-li">`,
   not StudyCard `<article>`. StudyCard root is `<div>`. Plan's
   `article[data-tags]` selector matches nothing on home. **Decision:**
   PR-1 explicitly specifies the per-page DOM contract: home uses
   `<li class="recent-li" data-tags="...">`, sites/date/tag-landing
   use StudyCard with the existing `<div>` root carrying `data-tags`.
   Filter logic queries `[data-tags]` regardless of element type.
6. **PR-1 + PR-2 = two precache waves.** Each PR rewrites the cached
   HTML, triggering full archive-cache revalidation twice. **Decision:**
   ship PR-1 + PR-2 in the same deploy (one merge, one push, one
   precache wave). Phasing them was wrong.
7. **bfcache hydrate race.** iOS Safari fires `pageshow` (with
   `event.persisted === true`) on bfcache restore, NOT
   `DOMContentLoaded`. **Decision:** PR-2 hydrates on both events;
   `form.reset()` before applying URL state ensures the rail matches
   the URL.
8. **Facet count semantics ambiguous.** Standard pattern: unchecked
   facets show "current ∪ this tag" count (would-be count if you ticked
   it); checked facets show current count. **Decision:** PR-2 implements
   this pattern; test spec explicitly asserts the math.

### P1 (must address before ship)

9. **Breakpoint ≥1024px geometrically invalid** vs existing rails at
   1200/1640. **Decision:** PR-2 layout solves filter rail + existing
   rails in one pass; PR-4 deleted (was collision repair).
10. **No `<link rel=canonical>` in `Base.astro`.** **Decision:** add
    canonical-link emission to Base.astro stripping all filter params
    in PR-1.
11. **Slug uniqueness assertion runs only in `build:day`.** **Decision:**
    add the assertion as an Astro `astro:build:done` hook so plain
    `astro build` also validates the lookup-table source.
12. **Built-intersection allowlist needed client-side** for the canonical
    redirect. **Decision:** PR-1 ships `<script type="application/json"
    id="built-intersections">` listing every generated `/tags/<a>+<b>/`
    URL; PR-5 client logic consults it.
13. **CF Analytics auto-tracks `history.replaceState`** → every chip
    click = pageview. **Decision:** PR-7 (observability) uses CF
    Analytics CUSTOM EVENTS not pageviews; pushState calls disable the
    auto-pageview via `dataset.cfBeacon`.
14. **Stale namespace map on cached pages.** **Decision:** version the
    inline `tag-namespaces` JSON (`{version: 1, map: {...}}`); namespace
    the URL form (`?tag=modality:radiation`) eliminating the lookup
    altogether; client falls back gracefully if version mismatches.
15. **iOS Safari 100vh + bottom-sheet drawer.** **Decision:** PR-3 uses
    `100dvh` + `pointer-events` shim; tested against iOS 18 specifically.
16. **`[hidden]` + dead scroll position.** **Decision:** PR-2 scrolls
    to the first visible card on filter change (preserves scroll only
    on initial URL-driven hydrate).

### P2 (polish)

17. `meeting` namespace missing from the mock checkbox tree — confirmed
    in scope; add to PR-2.
18. `Set.isSubsetOf` is Baseline-2024 (iOS 17.4+). **Decision:** PR-2
    uses an explicit loop instead.
19. Mobile drawer needs explicit dialog semantics (Escape, focus return,
    background inerting). **Decision:** PR-3 implements full WAI-ARIA
    dialog pattern.

## Codex review pass #2 amendments (2026-05-31)

Codex re-review against the amended plan surfaced 4 new P0s + 3 new P1s
the first amendment pass missed. Resolutions:

### P0 #1 — pushState cannot do cross-route canonicalization

On `/sites/breast/?tag=radiation`, pushing `/tags/radiation/` via
pushState would update the address bar without loading the tag page's
DOM. **Decision:** for cross-route canonical redirects, use
`window.location.assign(canonicalUrl)` (real navigation). Reserve
`history.replaceState` for SAME-ROUTE filter-state updates only.
PR-5 spec updated accordingly.

### P0 #2 + #3 — OR semantics dropped (cross-model tension resolved)

Expansion #4 (intra-namespace OR) directly conflicts with CEO finding
#2 (auto-redirect to canonical landing): there is no canonical
`/tags/radiation+surgery/` URL because intersection landings represent
AND only. RSS feeds same story. **User decision:** drop OR. Filter
stays strict AND across every namespace. Auto-redirect works for every
filter state. Trade-off accepted: readers wanting "radiation OR
surgery" must visit two tag pages separately. PR-2 spec reverted to
strict AND; Expansion #4 removed from accepted scope.

### P0 #4 — URL contract normalized

Single normative form: `?tag=<slug>` (repeated for multi-tag, no
namespacing). Slug-only is sufficient because v0.10's global slug-
uniqueness assertion guarantees no namespace ambiguity. **Decision:**
all references in the plan body, PR-2 spec, PR-5 spec, and test plan
read `?tag=<slug>` repeated. The earlier `?tag=modality:radiation`
namespaced form (P1 #14 amendment) is REVERTED — slug-only is simpler
and the slug-uniqueness invariant means it's safe.

Precedence rule: **URL params > saved-default localStorage > nothing**.
On every page load, the URL wins. Saved defaults apply only when the
URL has zero `?tag=` params.

### P1 #5 — PR-8 specified precisely

- Storage: `localStorage` (NOT cookies). Single key `oncbrain.savedFilters.v1`,
  payload `{tags: [...], savedAt: <iso8601>}`.
- Precedence: URL > saved > none (see P0 #4).
- Stale-tag handling: if a saved tag is no longer in the canonical
  enum (curator removed it), drop silently on load; surface a one-time
  banner the next reader visit.
- Clear behavior: "Clear all" in the filter UI clears both active
  filters AND saved defaults (one-button reset).
- Telemetry: hydration from saved defaults emits a `filter:hydrated`
  CF custom event with `source: "saved"` so PR-7 can distinguish
  saved-default hydrations from URL hydrations from user activations.

### P1 #6 — Analytics suppression sequencing

Codex correctly identified that PR-2 mutates history before PR-7
disables the auto-pageview. **Decision:** move the `dataset.cfBeacon`
suppression hook into PR-1 (alongside the canonical link emission),
so suppression is in place before any subsequent PR introduces
history mutations. PR-7 still owns the custom-event emission logic.

### P1 #7 — PR-10 `Esc` precedence

When the mobile drawer is open, `Esc` dismisses the drawer (standard
dialog pattern). When no drawer, `Esc` clears all filters. **Decision:**
spec'd in PR-10; precedence enforced by attaching the dialog-Esc
handler at a higher specificity than the filter-Esc handler. Shortcuts
`1-9` and `f` are NO-OPs when focus is in an editable control (input,
textarea, contenteditable).

### P2 #8 — Offline + nav test coverage

Add to PR-6 test matrix:
- PWA cached page survives `?tag=` param (SW normalization
  test from PR-1)
- Canonical real-navigation triggers SW route handler correctly
- Reload preserves URL state (URL > saved precedence)
- Back/forward across canonical redirects: `popstate` re-applies
  filter state from URL
